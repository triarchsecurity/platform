Required vs optional

What's actually required, and what isn't

The framework defaults to all the recommended pieces, but here's the honest breakdown of what your delivery pipeline must have, what we strongly suggest, and what's nice-to-have when capacity allows.

Required

Without these you don't have CI/CD

The minimum to call it a deployment pipeline.
  • GitHub repo with Actions enabledThe source of truth and the runner.
  • At least one CI workflowLint / test / one security scan on every PR.
  • At least one deploy workflowPushes to a target environment.
  • Branch protection on mainEven a basic ruleset blocking direct pushes.
  • Deploy credentialsOIDC or a scoped API token in env secrets.
  • A target hosting environmentWhere the React app actually runs.
Recommended

Default in this framework

Skip only with a documented reason.
  • 2+ protected environmentsDev + prod by default; add staging when team size or compliance scope warrants. verify-dev-deployed CI gate enforces promotion mechanically.
  • OIDC instead of long-lived tokensZero static cloud creds anywhere.
  • Required code review + CODEOWNERSBranch ruleset enforces review on sensitive paths.
  • Required signed commitsCloses the impersonation loophole.
  • SAST + SCA + secrets scanningSemgrep + OSV + Gitleaks at minimum.
  • IaC for cloud setupReproducible, code-reviewable, multi-account replicable.
  • Threat model in the repoEven a 1-page STRIDE table is better than none.
  • Audit log to immutable storageS3 Object Lock or equivalent.
Optional

Add when capacity allows

High value but not blocking initial deploy.
  • SLSA build provenance + cosignVerifiable artefact chain of custody.
  • Threat-model drift CI gateArchitecture change blocks merge until model updates.
  • DAST scanning (ZAP)Active web-app scan against dev/staging.
  • DR drill workflowQuarterly automated restore from prod backup.
  • Cost guardrail alertsNightly delta check; alert on >25% week-over-week.
  • License compliance checkBlock disallowed OSS licenses into prod.
  • Full observability stackGrafana / Datadog / Better Stack — GitHub gives you basics already.
  • Multiple SAST scannersCodeQL alongside Semgrep — fewer false negatives.
⚡

Shortcut path: GCP + Firebase

For an SMB React app where the customer has no compliance pressure and just needs to ship — Firebase is the simplest hosting target on the market. One Google account, one console, and you have hosting, auth, database, and (optionally) backend functions in ten minutes.

Trade-offs: vendor lock-in to Google; Firebase Auth lacks native SCIM (matters when selling to large B2B customers); cost can balloon at scale. Net for early-stage SMB: usually worth it. The walkthrough's cloud question includes "GCP + Firebase (simplest SMB path)" as a first-class option.

Environment model

Deploy flow

How to promote — dev, staging (optional), and prod

The 6 contract clauses (CL-1..CL-6) say WHAT must be true. This section says HOW you actually move code through each environment. Same branch-to-backend mapping for every project; the only choice is which UI/CLI you use to fire the prod promotion.

Branch → environment → backend

Branch Deploy fires when FAH backend Customer-visible host EnvBadge
feature/* never (PR only) — — —
dev PR merged → dev <app>-dev <app>-dev.<zone> DEV
staging (optional) PR merged → staging <app>-staging <app>-staging.<zone> STAGING
main PR merged → main + all 5 gates pass <app> <app>.<zone> none

Staging is optional. Every project ships with dev + main. Add staging when team headcount, compliance, or release-cadence-discipline warrants the third slot. Stage promotion mechanics mirror dev exactly; the staging → main PR is gated by cl4-gate the same way dev → main is.

DEPLOY TO DEV ~2-5 min wallclock · zero gates
  1. Cut a feature branch off dev: git checkout -b feat/<name> dev
  2. Push commits
  3. Open PR with base=dev
  4. CI runs quality-gate; merge when green (squash or merge commit — your call for feature → dev)
  5. Push to dev triggers deploy-dev job → deploys to <app>-dev FAH backend
  6. Container starts → ingests release_logs with env=dev → admin's version-snapshot reflects new dev version
  7. Customers see new code at <app>-dev.<zone> with the persistent DEV badge (CL-2)
Failure modes: CI fails (quality-gate) → PR can't merge. Deploy-dev fails (build error / secret missing) → dev backend keeps the previous version, admin doesn't update.
DEPLOY TO STAGING optional middle lane · stakeholder preview
  1. Branch off staging from dev (or hotfix branch): git checkout -b promote/<version> staging
  2. PR with base=staging
  3. Merge → push to staging triggers deploy to <app>-staging backend
  4. Customer-visible at <app>-staging.<zone> with STAGING badge (CL-2 renders for both dev and staging envs)
Use this lane to preview a release with stakeholders before opening the prod PR. Staging is the "customer-visible release-candidate" lane — same code that will ship to prod, on a separate URL with a visible badge.
DEPLOY TO PROD ~5-10 min wallclock · all 5 gates · NO BYPASS

Four equivalent paths — pick whichever your team prefers. All four converge on the same merge-to-main flow with the same gate chain.

TIER 1 Plain PR — zero new tooling (default for solo devs)
  1. Open PR with base=main head=dev (or staging if you use it)
  2. CI runs: quality-gate → version → cl4-gate → verify-dev-deployed (all must pass)
  3. Merge with "Create a merge commit" (NOT squash — the merge commit makes dev's tip a parent of main HEAD; verify-dev-deployed requires that ancestry)
  4. Push to main triggers deploy-prod → deploys to <app> backend
  5. Customers see new code at <app>.<zone> (no badge — prod chrome)
TIER 2 Slack — /triarch deploy (for teams in Triarch-managed Slack) 
/triarch deploy <project-key> <version>

OttoBot looks up the project, finds the release_logs row, dispatches promote-branch.yml on the project's repo (rebase dev → main → merge), posts the GH Actions run URL back to channel. Customer setup cost: 0.

TIER 3 CLI — gh workflow run (scriptable, no Slack) 
gh workflow run promote-branch.yml --repo <owner>/<repo> --ref main \
  --field branch=dev --field target_branch=main

Requires promote-branch.yml wrapper (~30-line workflow_dispatch referencing shared-workflows/promote-branch.yml@v4). Same end result as Tier 2.

TIER 4 Admin pipeline UI (Triarch-managed customers only)

Per-project staff page at admin.triarch.dev/admin/modules/pipeline/<slug> has a "Promote to production" button next to each approved RC. Two-step confirm modal → same dispatch path as Tier 2/3. Customer-facing release page at /projects/<slug>/releases has an Approve button that fires the same path.

No bypass. Every prod path runs the same 5 gates: quality-gate, version, cl4-gate (INV-1..5 + verdict POST), verify-dev-deployed, and CL-6 server-side. There is no force flag, no commit-message token, no admin override. The [hotfix-bypass-dev] token was removed 2026-05-14.

Hotfix path — same gates, faster cadence

There is no hotfix bypass. Even urgent fixes go through dev:

  1. Branch off main: git checkout -b hotfix/<bug> main
  2. Bump version (patch — e.g., v3.59.1)
  3. PR with base=dev → merge → dev backend deploys the hotfix (verify there)
  4. PR dev → main → merge with merge commit → prod deploys

Total wallclock: ~10 minutes including dev verification. The 2026-05-14 [hotfix-bypass-dev] removal was deliberate — if a fix needs to skip dev, the project's architecture is the problem, not the gate.

What blocks each promotion path

Gate When it fires What you do to unblock
quality-gate every PR fix the code (lint, test, build, or semgrep finding)
cl4-gate INV-1..5 every prod PR make sure dev has the exact version you're promoting (INV-4: target == dev_version). Wait 5 min if dev was just deployed (INV-5: dev age ≥ 300s).
verify-dev-deployed every prod push merge with "Create a merge commit", not squash. If you squashed, push main HEAD back to dev to restore ancestry.
CL-6 server-side every prod ingest if cl4-gate didn't post a verdict, admin's ingest returns 409 + audit row. Inspect the cl4-gate run logs.

Full spec: dev-prod-customer-contract.md "How to promote" section.

What's in the box

Capabilities, ranked by what auditors care about

Each one is enforceable code, not policy aspiration. If a control isn't a workflow file, environment rule, or branch ruleset entry — it isn't here.

Recommended

2 or 3 protected environments

Default: dev (auto) + prod (gated). Optional staging slot adds 1 reviewer + 5-min wait + soak time when team size or compliance scope justifies it. Prod gets 2 reviewers + 30-min cooling-off + tag-only on Enterprise plan; mechanical verify-dev-deployed gate enforces promotion on any plan.

Recommended

OIDC to your cloud

GitHub mints an ID token per job; cloud STS returns a short-lived session bound to repo:<org>/<repo>:environment:<env>. Zero static credentials. Scoped API tokens are an acceptable fallback if OIDC is too complex initially.

Recommended

7 required CI checks

lint-test · Semgrep · OSV-Scanner · Gitleaks · Checkov · tfsec · threat-model-drift. Critical/High findings block merge. SARIF uploaded to GitHub Security tab. Minimum required: at least one security scan.

Required

Branch protection on main

Minimum bar: block direct pushes, require PR review. Recommended: ruleset with signed commits · linear history · CODEOWNERS · conversation resolution · no force-push · no bypass actors.

Optional

SLSA provenance + cosign

Every container keyless-signed via Sigstore; every release carries a SLSA build-provenance attestation. Prod refuses unsigned artefacts. Real value at SOC 2 / supply-chain-conscious customers; defer otherwise.

Recommended

Audit log capture

Deploy events, approvals, security findings, admin actions emit structured events. Routed to immutable storage (S3 Object Lock or equivalent). GitHub keeps 90 days of run history natively as the baseline.

Optional

DR drill workflow

Quarterly scheduled job restores a prod backup into a throwaway staging stack. Produces an archived report. Skip until you have customers; add before you need it.

Optional

Cost guardrail

Nightly cost-API comparison. Alert + auto-issue if week-over-week delta > 25%. Cloud-provider native budgets are usually enough for early SMB.

Recommended

Threat model in repo

Even a 1-page STRIDE table is better than nothing. OPTIONAL upgrade: the Claude plugin adds STRIDE/OWASP/SOC2 mapping with file:line evidence and a CI drift gate.
