What attackers want

Three goals.
All in your GitHub.

Motive 1

Steal source code

IP, credentials hard-coded by mistake, customer data references, internal architecture, the .env that someone committed in 2021.

Motive 2

Inject malicious commits

Supply-chain attack. Compromise one dev's PAT, push a sneaky workflow change, ship to your customers' production.

Motive 3

Pivot to your cloud

Workflows have OIDC trust. Compromise the repo, the workflow runs, the cloud session is yours for an hour.

The vector is rarely "hack the source code." It's "compromise an account, push a commit, deploy."

Act 2 · Identity

Two-factor. Required.
Phishing-resistant if you can.

Not all 2FA is equal. SMS gets SIM-swapped. TOTP can be phished. Hardware keys and passkeys can't.

Act 2 · How automation authenticates

Password → PAT → Fine-grained PAT → GitHub App.

Four ways to talk to the GitHub API. Each step up is more secure, more auditable, more rotation-friendly.

Auth methodUse forRiskRotation
Password + 2FAHumans logging inPhishingQuarterly
Classic PATLegacy CI · don't use in 2026High — broad scopes, hard to auditPer leak (often years)
Fine-grained PATPersonal scripts, narrow accessLower — specific repos + permissionsRequired ≤1 year, customizable
GitHub AppOrg-wide automationLowest — installable, auditable, scopedAuto-rotated · 1-hour tokens

Move automation up the ladder. Don't ship a classic PAT in 2026.

Act 2 · How git itself authenticates

SSH for git push.
Tokens for the API.

# SSH — most common for human developers ssh-keygen -t ed25519 -C "you@example.com" # Add ~/.ssh/id_ed25519.pub to your GitHub account git remote set-url origin git@github.com:org/repo.git # HTTPS + PAT — for environments that block SSH (corporate proxies) git remote set-url origin https://github.com/org/repo.git # Then `git push` prompts for username + PAT (use a fine-grained one)

When a developer's laptop is lost, revoke their SSH keys + PATs immediately from their GitHub account settings. Then enforce 2FA on re-login.

Act 3 · Permissions

Teams, then roles.
Not the other way around.

Organization
└── Default base permission: None (NOT "Read")
Teams
├── @org/engineering → granted Write on app repos
├── @org/release-managers → granted Maintain on app repos
└── @org/security → granted Maintain on .github + iac paths
Roles (least → most)
Read · Triage · Write · Maintain · Admin

Direct user-to-repo grants are a maintenance trap. Use teams. Audit your default base permission — it's "Read" out of the box, which is too generous.

Act 4 · Automatic reviewers

CODEOWNERS — auto-route reviews to the right team.

# .github/CODEOWNERS # default — engineering reviews everything * @org/engineering # CI/CD pipeline itself — security AND release-managers must review /.github/ @org/security @org/release-managers /.github/workflows/ @org/security @org/release-managers /iac/ @org/security @org/release-managers # auth code — extra scrutiny /src/auth/ @org/security **/secrets* @org/security

Pair with require_code_owner_review: true in the ruleset. Now the right humans have to approve, not just see.

Act 4 · Provenance

Cryptographic identity
on every commit.

Otherwise: anyone with push access can write commits in your name. The author field is just a string.

# One-time setup — generate a signing key (or use an existing GPG/SSH one) ssh-keygen -t ed25519 -f ~/.ssh/git_signing_key # Tell git to sign with it git config --global user.signingkey ~/.ssh/git_signing_key.pub git config --global gpg.format ssh git config --global commit.gpgsign true # Add the public key to GitHub as a "signing key" gh ssh-key add ~/.ssh/git_signing_key.pub --title "work-laptop-signing" --type signing

Now every commit shows a Verified badge. The ruleset rejects unsigned ones.

Act 5 · Secrets

Three scopes, plus Dependabot.

Pick the narrowest scope that works. Auditing unused org secrets quarterly catches drift.

Repo secret

This repo's workflows only

The default. Use unless you have a reason not to.

Environment secret

Only when deploying to that env

Scoped and gated by environment protection rules. Use for production credentials.

Org secret

Multiple repos can read

Restrict to specific repos. Useful for things like a shared Slack webhook.

Dependabot secret

Only Dependabot updates

Lives in a separate slot — Actions workflows can't read these. Use for private package registries.

Repo Settings → Secrets and variables → Actions / Dependabot · Org Settings → Secrets → Restrict to selected repositories
Act 5 · Catch them before merge

Push protection blocks secrets at the source.

When secret scanning sees an AKIA-pattern in a push, it rejects the push. Developer fixes locally. No leaked secret in the git history.

remote: error: GH013: Repository rule violations found remote: -- A potential secret was detected: remote: Type: AWS Access Key ID remote: Path: src/config.js:14 remote: Push declined. # Fix locally: git rm src/config.js git commit --amend git push
Repo Settings → Code security → Secret scanning → Push protection: Enabled

Free for public repos. Included with GitHub Advanced Security on private. Worth the spend.

Act 6 · Code & supply chain

Keep deps fresh.
Catch vuln patterns.

Dependency hygiene

Dependabot

Weekly PRs to update vulnerable transitive deps. Free, built-in. Configure ecosystems in .github/dependabot.yml.

Static analysis

CodeQL

Semantic SAST. Runs the security-extended queries by default. Free for public, GHAS for private.

# Minimal CodeQL workflow on: [push, pull_request] jobs: analyze: runs-on: ubuntu-latest permissions: { security-events: write, contents: read } steps: - uses: github/codeql-action/init@v3 with: { languages: javascript-typescript, queries: security-extended } - uses: github/codeql-action/autobuild@v3 - uses: github/codeql-action/analyze@v3
Act 6 · The GITHUB_TOKEN scope

Default to read-only.
Opt in to write.

GITHUB_TOKEN is auto-injected into every workflow. By default it has write access to the entire repo. Flip the default org-wide.

permissions: { contents: read } # at the top of every workflow file jobs: build: # Each job opts in to what it needs: permissions: contents: read packages: write # to push containers to GHCR id-token: write # to mint OIDC token for cloud auth attestations: write # to publish SLSA attestations permissions: write-all # NEVER. This is what attackers exploit.
Org Settings → Actions → General → Default GITHUB_TOKEN permissions: Read repository contents and packages permissions
Act 7 · Actions security

Allowlist what runs in your CI.

By default, any GitHub Action from any author can run in your workflows. That's a supply-chain firehose. Lock it down.

# Minimal allowlist for the Triarch framework github_owned_allowed: true # actions/* always OK verified_allowed: false # GitHub's "verified creator" bar is too low patterns_allowed: - semgrep/*@* - aquasecurity/trivy-action@* - aquasecurity/tfsec-*@* - anchore/*@* - google/osv-scanner-action@* - gitleaks/gitleaks-action@* - bridgecrewio/checkov-action@* - sigstore/cosign-installer@* - slsa-framework/*@* - hashicorp/setup-terraform@* - opentofu/setup-opentofu@*
Org Settings → Actions → General → Allow specific actions and reusable workflows

If a marketplace action isn't on the list, it can't run. Workflows fail at parse time, not at runtime.

Act 7 · The 2026 lesson

SHA pin third-party actions.
Always.

Tags can be moved. SHAs can't. In March 2026 attackers hijacked 75 of 76 tags on aquasecurity/trivy-action to steal cloud creds from 10,000+ workflows. SHA-pinned workflows were unaffected.

- uses: aquasecurity/trivy-action@v1 # tag — mutable - uses: aquasecurity/trivy-action@latest # worse - uses: aquasecurity/trivy-action@692973e3d937129bcbf40652eb9f2f61becf3332 # SHA — immutable # v1.7.0 ↑ kept in trailing comment

Dependabot understands the trailing version comment. It bumps the SHA + comment together. Pin once, automate forever.

Org policy: enforce SHA pinning via Org Settings → Actions → General → Allow specific actions → Require pinning
Act 7 · Cloud auth

OIDC beats long-lived API keys.

A workflow that ships an AWS access key in a secret is one leaked log line away from disaster. OIDC removes the secret entirely.

permissions: { id-token: write, contents: read } jobs: deploy: runs-on: ubuntu-latest environment: prod steps: - uses: aws-actions/configure-aws-credentials@v4 with: role-to-assume: arn:aws:iam::ACCT:role/github-actions-prod aws-region: us-east-1 # Now `aws ...` works for the next ~1 hour. No static keys anywhere.

Zero static cloud keys = zero key-rotation cron jobs = zero leaked-key incidents.
Trust policy on the IAM role binds repo:org/repo:environment:prod — no other repo can assume it.

Act 8 · Who did what

Every action logged.
Stream it to S3.

# Sample audit log events GitHub emits: {"action": "team.add_member", "actor": "alice", "team": "admin", "member": "bob"} {"action": "protected_branch.policy_override", "actor": "carol", "branch": "main"} {"action": "oauth_authorization.create", "actor": "dave", "app": "unknown-3rd-party"} {"action": "repo.access", "actor": "erin", "repo": "prod-secrets"} {"action": "personal_access_token.access_granted", "actor": "frank", "scopes": ["admin:org"]}
Free / Team

90 days, query-only

Use the audit-log API. Schedule a workflow to export weekly to your own bucket if you want longer retention.

Enterprise

Streaming → S3 / Splunk / Datadog

Real-time, indefinite retention, SIEM-ready. Set up: Org → Settings → Audit log → Streaming.

If you can't reconstruct who did what 6 months later, you don't have audit.

Act 10 · The checklist

Eight settings that move you
from "it works" to "it's hardened."

1
Org-wide 2FA requiredSettings → Authentication security → "Require two-factor authentication for everyone." Passkeys or hardware keys preferred.
2
Default base permission = NoneSettings → Member privileges. Anything else means new members get reads on everything.
3
Restrict who can create classic PATsSettings → Personal access tokens. Block classic PATs entirely if you can; require fine-grained.
4
Branch ruleset on every important repoRequired signed commits, CODEOWNERS review, status checks, linear history. No bypass actors.
5
Allowed actions allowlist + SHA pinningOrg Settings → Actions → General. Curate the allowlist. Enforce SHA pinning policy.
6
OIDC for every cloud authDelete static AWS_ACCESS_KEY_ID secrets. Use federated identity per environment.
7
Push protection on Secret ScanningRepo Security → Secret scanning. Catches secrets at push time, not at PR review time.
8
Audit log streaming (or weekly export)Enterprise: stream to S3 / Splunk. Team: schedule a workflow that pulls from the audit-log API.
