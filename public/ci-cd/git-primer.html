Act 1 · Setup

Git is the engine.
GitHub is the highway.

⚙️
The tool

Git

A program you run on your laptop. Version-controls files. Works offline. Tracks who changed what, when, why. Created by Linus Torvalds in 2005.

🌐
The service

GitHub

A website that hosts your git repos remotely. Adds collaboration: pull requests, code review, issues, Actions, identity. You don't need it to use git — but at scale you do.

Two different things. You install git locally. You authenticate to GitHub remotely. Git pushes commits to GitHub; GitHub renders them.

Mental model

Three places your code lives.

Every file in a git repo is in one of three "trees" at any moment. Most confusing git errors come from forgetting which tree you're in.

1. Working directory

The actual files on disk. What your editor sees. Anything you've changed but not yet staged.
edit a file

2. Staging area

A list of "I want this change in the next commit." Lets you prepare a coherent commit before making it.
git add <file>

3. Repository

Permanent history. Once committed, this snapshot is identified by a SHA and can never be lost (unless you really try).
git commit

git status tells you what's in each tree right now. Run it constantly — it's git's "where am I" command.

Act 2 · Your first contribution

Step 1 — clone the repo.

A clone is a complete copy of the repo — every commit, every branch, every tag. You can work offline. You push changes back when ready.

# Clone over SSH (recommended for daily work) git clone git@github.com:acme-corp/web-app.git cd web-app # See what you got git log --oneline -5 # a1b2c3d (HEAD -> main, origin/main) feat: add user login # e4f5g6h fix: typo in README # ... git branch -a # list all branches git remote -v # where does this clone push to?
Act 2 · The atom of git

Edit. Stage. Commit.

A commit is a snapshot — not a diff. It records the entire state of all tracked files at one moment, plus an author, a message, and a parent commit. Identified by a 40-char SHA.

# 1. Edit some files vim src/auth.js # 2. Stage what you want in the next commit git add src/auth.js # single file git add -p # interactive — stage individual hunks git add . # everything (be careful) # 3. Commit (snapshot + message) git commit -m "feat(auth): handle expired tokens on refresh" # Or open an editor for a multi-line message: git commit
Act 2 · Commits future-you will love

Write conventional commit messages.

Format: type(scope): subject. Type is one of feat fix docs chore refactor test perf. Body explains why, not what (the diff already shows what).

BadGoodBetter
✕ fix stuff ~ fix: handle expired tokens ✓ fix(auth): refresh tokens before expiry
✕ changes ~ refactor: split user.js ✓ refactor(user): extract validation to user-rules.js
✕ wip ~ feat: add login ✓ feat(login): support OAuth via Google

Bonus: tools like semantic-release read these to auto-version your releases. Good commits become free changelogs.

Act 2 · Share what you've done

Step 3 — push to the remote.

Until you push, your commits exist only on your laptop. Push uploads them to GitHub so collaborators can see them.

# First time pushing a new branch: git push -u origin feature/login # -u sets the upstream so future `git push` and `git pull` know where # After that, just: git push # See what's about to be pushed before doing it: git log origin/feature/login..feature/login --oneline # Force push? Almost never. Use --force-with-lease if you must: git push --force-with-lease # refuses if remote has commits you don't have
Act 3 · Branches

A branch is just a pointer.

Not a folder. Not a copy of the code. Just a label that points to a commit. When you commit on a branch, the label automatically moves to the new commit.

# A repo with two branches looks like this: main: A───B───C───D ← main points here \ feature: E───F ← feature points here # Both branches share commits A, B, C, D # feature has its own commits E, F that main doesn't know about # main and feature are just two labels on a shared commit graph

This is why creating a branch is instant — it's just writing a name into a tiny file. HEAD is the special pointer to "which branch you're on right now."

Act 3 · Branch operations

Create. Switch. Delete.

# Create a new branch off your current commit + switch to it git checkout -b feature/login # classic syntax git switch -c feature/login # modern (clearer intent) # Switch between existing branches git switch main git switch feature/login # Delete a local branch (only if its commits are merged or you don't care) git branch -d feature/login # safe — refuses if unmerged git branch -D feature/login # force — careful # Delete the remote branch too git push origin --delete feature/login # See what you've got git branch # local branches git branch -r # remote-tracking branches
Act 3 · Pick a model

Trunk-based or git-flow.

Recommended for most teams

Trunk-based

main plus short-lived feature branches that live ≤2 days. Merge often. Delete the branch after.

Branches: main, feature/*, hotfix/*

Heavy ceremony

Git-flow

main, develop, release/*, hotfix/*, feature/*. Designed for shrink-wrapped releases. Most modern teams find it over-engineered.

Branches: 5 long-lived types

Pick trunk-based unless you're shipping desktop software with quarterly releases. Most SaaS teams want fast merges and a single source of truth — main.

Act 4 · Pull requests

A PR is an offer to merge.

You're saying "here's my work on branch X — please review, then merge it into main." GitHub gives reviewers a UI to comment line-by-line, request changes, and approve.

# Open a PR after pushing your branch gh pr create --base main --head feature/login --title "feat: user login" --body "Adds OAuth via Google + GitHub" # Or via the GitHub UI: visit the repo, GitHub auto-detects your pushed branch # and offers a big green "Compare & pull request" button. # What a PR contains: - The diff between your branch and the target branch (usually main) - The commits you've added on this branch - A title + description (think: the merge commit message) - Reviewers, assignees, labels, linked issues, milestones - Status checks (CI results, security scans, deploy previews) - A discussion thread (comments, approvals, change-requests)

Act 4 · Automation as a reviewer

CI protects what humans miss.

"Required status checks" make merging impossible until specific automated checks pass. Tests, linting, security scans — anything you want enforced.

# What a PR's check summary looks like: ✓ lint # 12s ✓ unit-tests # 1m 24s ✓ semgrep # 38s ✓ osv-scanner # 22s ✗ codeql # found 2 High-severity findings ← blocks merge ✓ build # 3m 02s # Merge button is DISABLED until codeql is green.

Without required checks, anyone with merge permission can ignore CI. With required checks, even admins go through the gate.

Act 5 · The core decision

Three ways to merge.
Different histories.

GitHub lets you pick the merge strategy per PR (or enforce one repo-wide). Each leaves a different shape in your commit graph.

Strategy 1

Merge commit

Preserves the full branch topology. Creates a "merge commit" that ties the two histories together.

Strategy 2 · most common

Squash and merge

Collapses all the PR's commits into one clean commit on main. Loses granular detail but keeps main tidy.

Strategy 3

Rebase and merge

Rewrites the PR's commits onto main, no merge commit. Linear history. Rewrites SHAs.

Strategy 1 · Merge commit

Preserves branch topology.

# Before merge: main: A───B───C───D \ feature: E───F───G # After "Merge pull request": main: A───B───C───D────────M ← M is the merge commit \ / feature: E───F───G # Pros: complete history, easy to see "this came from a PR" # Cons: messy graph at scale, many merge commits = noise

Use when: long-running feature branches where the individual commits genuinely matter for debugging or git-blame archaeology.

Strategy 2 · Squash and merge

Many WIP commits → one clean commit.

# Before: main: A───B───C───D \ feature: E───F───G # E = "wip", F = "fix tests", G = "address review" # After "Squash and merge": main: A───B───C───D───S ← S = single commit with E + F + G's combined diff # The PR title becomes S's commit message. # E, F, G are gone from main. They still exist on the feature branch (if you didn't delete it).

Use when: most PRs. Feature branches accumulate ugly commits ("wip", "fix typo", "address review feedback") that aren't worth keeping in main's history. Squash collapses them to one clean line.

Recommended default for most teams.

Strategy 3 · Rebase and merge

Linear history, no merge commits.

# Before: main: A───B───C───D \ feature: E───F───G # After "Rebase and merge": main: A───B───C───D───E'──F'──G' ← commits replayed onto main; SHAs change # Pros: dead-straight history. `git log --oneline` is one column. # `git bisect` is easier. No diamonds in the graph. # Cons: SHAs rewritten. NEVER rebase branches that others have based work on. # Lose the "these commits arrived together as PR #123" signal.

Use when: small, well-crafted feature branches where each commit is meaningful and the team agrees on linearity. Common in compiler / kernel / infrastructure teams; less common in product teams.

Act 6 · When merges aren't automatic

A conflict = two changes,
same line.

Git tries to auto-merge. When it can't decide between two competing changes to the same line, it stops and leaves the file with conflict markers.

// src/config.js — file with a conflict <<<<<<< HEAD # your branch's version const port = 3000; ======= const port = process.env.PORT || 8080; # the incoming branch's version >>>>>>> main # Resolve by editing the file — keep whichever you want (or combine): const port = process.env.PORT || 3000; # merge of both intents # Then mark it resolved + continue: git add src/config.js git commit # for merge — auto-fills message git rebase --continue # for rebase
Act 6 · Getting others' work

pull = fetch + merge.

# `pull` is two commands in one: git pull # = `git fetch origin` + `git merge origin/<branch>` # If you've made local commits and main moved while you were working, # the default `git pull` creates a "merge of main into your branch" commit. # That's noise. Prefer rebase: git pull --rebase # fetches, then replays YOUR commits on top of theirs. Linear, no diamond. # Make this the default once: git config --global pull.rebase true # For pure visibility without changing anything: git fetch # downloads remote refs but doesn't touch your working dir git log HEAD..origin/main --oneline # what's on main that you don't have?
Act 6 · Three ways to take it back

amend, revert, reset.
Pick the right one.

You want to…UseSafe after push?
Fix the last commit's message or contents git commit --amend ✕ No — rewrites the last commit's SHA
Undo a commit that's already pushed / merged git revert <sha> ✓ Yes — creates a new commit that inverts the old one
Throw away local commits entirely git reset --hard <sha> ✕ NEVER on shared branches — destroys history
Move HEAD back but keep changes staged git reset --soft <sha> ✕ Same caveat — local only
Recover a commit you thought you lost git reflog + git reset --hard <sha> ✓ reflog tracks every move HEAD makes for 90 days

Rule of thumb: if the commit is pushed/merged, use revert. If it's still local, you have more freedom. reflog is the safety net most people don't know exists.

Act 7 · GitHub-specific units of work

Issues, PRs, Forks, Projects.

🐛
Track work

Issues

Bug reports, feature requests, tasks. Labelable, assignable, linkable to PRs. The atomic unit of "we need to do something."

🔀
Ship work

Pull requests

Code review + status checks + merge gate. Closes issues automatically via Closes #42 in the body.

🍴
Open-source contribution pattern

Forks

Your own copy of someone else's repo. Push to your fork; open a PR from your fork to the original. Standard for OSS contributions.

📋
Organize work

Projects

Kanban + spreadsheet hybrid for grouping issues and PRs. Replaces JIRA at small scale. Per-org or per-repo.

Issues are requests. PRs are responses. Forks are independent copies. Projects are views over issues + PRs.
