<decisions>
## Implementation Decisions

### What this phase does NOT do
- Does NOT autonomously add DNS records via GoDaddy MCP. The fah-claim TXT + _acme-challenge CNAME values are generated by Firebase Console at the moment of "Add custom domain" — must be copy-pasted from Console into GoDaddy at that step. Pre-adding A records is technically possible but unnecessary (Console does the full set).
- Does NOT create FAH backends. For triarch.dev hostnames, backends already exist. For triarchsecurity.com hostnames, Phase 33/34 creates the backends — only AFTER those phases ship can Phase 30 finish.

### What this phase DOES (for the human)
Provides:
- A precise list of the 6 hostnames + which Firebase project + which FAH backend each maps to
- The expected DNS record pattern (A + TXT + CNAME per host, mirroring the existing darksouls-dev entries)
- The exact Firebase Console URLs to click "Add custom domain" in
- Verification commands (curl, dig, openssl) for each hostname post-claim
- Updated apphosting.dev.yaml NEXTAUTH_URL values where applicable

### Existing DNS records as reference template
darksouls-dev (already working) has these 3 record types in GoDaddy:
1. `A` record: `darksouls-dev` → `35.219.200.0` (TTL 600)
2. `TXT` record: `darksouls-dev` → `fah-claim=002-02-41876c96-dff1-4285-b05d-b76684819bb7`
3. `CNAME` record: `_acme-challenge_rvlyulquknfzxvy4.darksouls-dev` → `<unique>.authorize.certificatemanager.goog`

Each new hostname follows this 3-record template. The fah-claim TXT value and the _acme-challenge CNAME value are unique-per-hostname — generated by Firebase Console.
