<objective>
Refactor `promoteAndAudit()` so the web Promote route can call it without Slack message coordinates. The function currently requires both `channelId` and `messageTs` (Slack thread_ts) to post a threaded reply and chat.update the original message. The web flow has neither — there's no Slack message to thread onto.

Resolution per CONTEXT.md decisions: make both fields nullable. When null, skip the threaded-reply path AND skip chat.update; instead, post a fresh standalone Slack message to the project's release-approvals channel announcing the promotion (so Slack is always informed about a promotion, regardless of origin). When non-null, behavior is identical to today.

This is the design decision called out in STATE.md Pending Todos ("resolve promoteAndAudit signature for web context"). Resolved as Option (a) — nullable params with fallback notification. Option (b) (split into dispatchPromotion + notifySlack) was considered but rejected because it forces every caller to remember two function calls; the unified function with origin-aware branching is simpler and harder to misuse.

Purpose: Unblocks plan 09-03 (web Promote API route) and satisfies PROM-03 (Slack notification posted on every successful dispatch — web or Slack origin).
Output: Backwards-compatible promoteAndAudit + green Vitest suite covering both origins.
</objective>
