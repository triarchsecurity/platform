<objective>
Build the staff-only Web Promote API route with an atomic UPDATE-based race guard on `release_logs.promotion_dispatched_at`. Mirror the same atomic guard on the Slack OttoBot wantsApprove handler so the web and Slack staff-promote paths can NEVER both fire `promoteAndAudit` for the same release. Additionally, populate `actor_source` on the customer's existing `release_approvals` row by threading an optional `actorSource` parameter through `approveRelease` and `rejectRelease` (Pitfall 4 audit-trail unification).

The atomic UPDATE-with-WHERE-IS-NULL pattern is the actual idempotency mechanism for the staff-dispatch race: `UPDATE release_logs SET promotion_dispatched_at=now(), promotion_dispatched_by=$email WHERE id=$id AND promotion_dispatched_at IS NULL RETURNING ...`. Exactly one concurrent caller gets a row back (race won → proceeds to promoteAndAudit); the other gets an empty result (race lost → returns 409 web / ephemeral Slack message).

Why not the partial unique index from plan 09-01? That index guards a different race — multiple `decision='approved'` rows on `release_approvals` (the customer-approve race). Staff-dispatch attempts do not write `release_approvals` rows; they update `release_logs`. The two race-guards are orthogonal and both valuable: 09-01's index is defense-in-depth for the customer-approve flow; this plan's atomic UPDATE is the staff-dispatch race-guard.

Purpose: PROM-03, PROM-04, PROM-05 — Web Promote dispatches via promoteAndAudit, race-guarded against concurrent Slack dispatch, with inline result for the in-flight UI.
Output: Working POST /api/admin/releases/[id]/promote with full test coverage; Slack handler matched with the same atomic guard; release-actions.ts populates actor_source.
</objective>

    Create `src/app/api/admin/releases/[id]/promote/route.ts`:

    ```typescript
    import { NextRequest, NextResponse } from 'next/server';
    import { db } from '@/lib/db';
    import { releaseLogs } from '@/db/schema';
    import { and, eq, isNull, sql } from 'drizzle-orm';
    import { requireStaff } from '@/lib/api-auth';
    import { promoteAndAudit } from '@/lib/release-promotion';

    export async function POST(
      req: NextRequest,
      { params }: { params: Promise<{ id: string }> },
    ) {
      // 1. Auth
      const auth = await requireStaff();
      if (auth.error) return auth.error;
      const actorEmail = auth.session!.user!.email!;

      // 2. Resolve release
      const { id } = await params;
      const [release] = await db.select().from(releaseLogs).where(eq(releaseLogs.id, id));
      if (!release) {
        return NextResponse.json({ error: 'not_found' }, { status: 404 });
      }

      // 3. Status guard
      if (release.status !== 'approved') {
        return NextResponse.json(
          { error: 'invalid_status', currentStatus: release.status ?? 'dev' },
          { status: 400 },
        );
      }

      // 4. Atomic race guard via conditional UPDATE
      const [claimed] = await db
        .update(releaseLogs)
        .set({
          promotionDispatchedAt: new Date(),
          promotionDispatchedBy: actorEmail,
        })
        .where(and(eq(releaseLogs.id, id), isNull(releaseLogs.promotionDispatchedAt)))
        .returning({
          id: releaseLogs.id,
          promotionDispatchedAt: releaseLogs.promotionDispatchedAt,
          promotionDispatchedBy: releaseLogs.promotionDispatchedBy,
        });

      if (!claimed) {
        // Lost the race — re-read to surface who beat us
        const [current] = await db
          .select({
            promotionDispatchedAt: releaseLogs.promotionDispatchedAt,
            promotionDispatchedBy: releaseLogs.promotionDispatchedBy,
          })
          .from(releaseLogs)
          .where(eq(releaseLogs.id, id));
        return NextResponse.json(
          {
            error: 'already_promoted',
            dispatched_by: current?.promotionDispatchedBy ?? null,
            dispatched_at: current?.promotionDispatchedAt?.toISOString() ?? null,
          },
          { status: 409 },
        );
      }

      // 5. Dispatch (NOT fire-and-forget for web — caller wants the result inline for PROM-05)
      const result = await promoteAndAudit({
        release,
        actorEmail,
        channelId: null,
        messageTs: null,
        slackUserName: null,
      });

      if (!result.ok) {
        return NextResponse.json({ ok: false, error: result.error }, { status: 200 });
      }
      return NextResponse.json({ ok: true }, { status: 200 });
    }
    ```

    Key design points:
    - The atomic UPDATE-with-WHERE pattern uses `isNull(promotionDispatchedAt)` as the race guard. This is the actual idempotency mechanism for the staff-dispatch race; it does not depend on the partial unique index from plan 09-01 (which guards customer-approve race instead).
    - `promoteAndAudit` is awaited (not fire-and-forget) so the route can return success/failure inline for PROM-05's "in-flight then surfaces result" UX.
    - Slack 3-second rule does NOT apply here — this route is hit from the staff browser, not from Slack. Browser tolerates the few-second dispatch latency; the client-side spinner (plan 09-05) covers the wait.
    - On dispatch failure we return HTTP 200 with `{ ok: false, error }` because the request was processed; the audit row was already updated by promoteAndAudit. The client island treats `ok: false` as the failure signal and surfaces the error pill linking to the Actions run URL (PROM-05).
