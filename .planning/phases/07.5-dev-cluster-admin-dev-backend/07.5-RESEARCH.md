**Critical lesson:** Do NOT tag v4 in the same plan that writes the v4 files. Tag AFTER canary is verified. Phase 2-03 set this precedent; v4 must follow it.

### Pattern 2: HUMAN-UAT Runbooks

Every phase with mandatory human action emits an HTML runbook (Phase 1 vault, Phase 3 GitHub App, Phase 7 Slack scope, Phase 7.5). The HTML file is self-tracking with checkboxes and is stored at `.planning/phases/<phase>/07.5-RUNBOOK.html`. This is already done — the RUNBOOK.html exists. The code plans do not reproduce runbook steps; they reference the RUNBOOK.html.

### Pattern 3: `${{ }}` in env: blocks (not run: steps)

STATE.md: "Phase 02-02: All ${{ }} expressions moved to env: blocks in run: steps — required for actionlint/shellcheck compliance." All new YAML in db-migrate.yml must follow this pattern: set `env:` keys from `${{ inputs.* }}` and `${{ secrets.* }}`, then reference via `$VARIABLE_NAME` in bash.

### Pattern 4: continue-on-error: true for callbacks

All admin callbacks (deploy-firebase.yml Notify Admin step, deploy-prod.yml Notify Admin step, promote-branch.yml callback job) use `continue-on-error: true`. db-migrate.yml should NOT follow this for the drizzle push step — a failed migration should fail the workflow (that's the entire point). The empty-token guard does follow this pattern.

### Pattern 5: Two-step Secret Access for drizzle-kit push

Current manual process (onboarding-projects.md Step 8 verify block):
```bash
DATABASE_URL=$(firebase apphosting:secrets:access DATABASE_URL --project triarch-dev-website)
psql "$DATABASE_URL" -c "..."
```

`firebase apphosting:secrets:access <SECRET> --project <fb_project_id>` returns the secret value to stdout. db-migrate.yml replaces this with a CI-automated step: read secret → export as env var → run drizzle-kit push.

**Auth required:** `firebase` CLI in CI authenticates via `FIREBASE_SA_KEY` (same SA used in deploy-firebase.yml). The SA must have `secretmanager.secretAccessor` on the target secret in the Firebase project.

**Note on git_branch for dev:** When env=dev, the `git_branch` input should default to `dev` (not `main`). However, the existing `git_branch` input default is `'main'`. Two options for v4:

Option 2 preserves backwards compat for callers who explicitly pass `git_branch=main`. The step should use: `DEPLOY_BRANCH="${GIT_BRANCH:-main}"; if [ "$ENVIRONMENT" = "dev" ] && [ "$GIT_BRANCH" = "main" ]; then DEPLOY_BRANCH="dev"; fi`

### db-migrate.yml (complete new workflow)

```yaml
# shared-workflows/.github/workflows/db-migrate.yml
# Reusable workflow: run drizzle-kit push against the dev or prod database.
# Replaces the manual ritual:
#   firebase apphosting:secrets:access DATABASE_URL --project X | psql -f migration.sql
#
# Triggered via workflow_call from consumer ci-cd.yml OR workflow_dispatch (manual one-shot).
# WORKFLOW-04 decision: manual workflow_dispatch only for v2.0; auto-trigger on schema.ts diff is v3.

name: DB Migrate

on:
  workflow_call:
    inputs:
      environment:
        description: 'dev or prod'
        required: false
        type: string
        default: 'dev'
      firebase_project_id:
        description: 'Firebase project ID to read the database secret from.'
        required: true
        type: string
      consumer_repo:
        description: 'Consumer repo (owner/repo) to checkout for drizzle config.'
        required: true
        type: string
      consumer_ref:
        description: 'Git ref to checkout. Defaults to environment branch (dev for dev, main for prod).'
        required: false
        type: string
        default: ''
    secrets:
      FIREBASE_SA_KEY:
        description: 'GCP service-account JSON with Firebase secret access.'
        required: true
      GH_PAT:
        description: 'PAT to checkout consumer_repo if private. Falls back to github.token.'
        required: false
  workflow_dispatch:
    inputs:
      environment:
        description: 'dev or prod'
        required: false
        type: string
        default: 'dev'
      firebase_project_id:
        description: 'Firebase project ID (e.g. triarch-dev-website for admin dev)'
        required: true
        type: string
      consumer_repo:
        description: 'Consumer repo (e.g. MyAlterLego/triarch-dev)'
        required: true
        type: string
      consumer_ref:
        description: 'Git ref to checkout (leave blank for default)'
        required: false
        type: string
        default: ''

permissions:
  contents: read
  packages: read

jobs:
  migrate:
    name: drizzle-kit push (${{ inputs.environment }})
    runs-on: ubuntu-latest
    timeout-minutes: 10

    steps:
      - name: Checkout consumer repo
        uses: actions/checkout@v4
        with:
          repository: ${{ inputs.consumer_repo }}
          token: ${{ secrets.GH_PAT || github.token }}
          ref: ${{ inputs.consumer_ref != '' && inputs.consumer_ref || (inputs.environment == 'dev' && 'dev' || 'main') }}

      - name: Set up Node.js
        uses: actions/setup-node@v4
        with:
          node-version: '22'
          registry-url: 'https://npm.pkg.github.com'
          cache: 'npm'

      - name: Install dependencies
        run: npm ci
        env:
          NODE_AUTH_TOKEN: ${{ secrets.GH_PAT || github.token }}

      - name: Install Firebase CLI
        run: npm install -g firebase-tools

      - name: Authenticate to Google Cloud
        uses: google-github-actions/auth@v2
        with:
          credentials_json: ${{ secrets.FIREBASE_SA_KEY }}

      - name: Set up Cloud SDK
        uses: google-github-actions/setup-gcloud@v2

      - name: Read database secret
        id: read_secret
        env:
          ENVIRONMENT: ${{ inputs.environment }}
          FIREBASE_PROJECT_ID: ${{ inputs.firebase_project_id }}
        run: |
          if [ "$ENVIRONMENT" = "dev" ]; then
            SECRET_NAME="DATABASE_URL_DEV"
          else
            SECRET_NAME="DATABASE_URL"
          fi
          echo "Reading $SECRET_NAME from project $FIREBASE_PROJECT_ID"
          DB_URL=$(firebase apphosting:secrets:access "$SECRET_NAME" --project "$FIREBASE_PROJECT_ID")
          # Write to masked output — GitHub Actions masks values set via ::add-mask::
          echo "::add-mask::$DB_URL"
          echo "database_url=$DB_URL" >> "$GITHUB_OUTPUT"

      - name: Run drizzle-kit push
        env:
          DATABASE_URL: ${{ steps.read_secret.outputs.database_url }}
          ENVIRONMENT: ${{ inputs.environment }}
        run: |
          echo "Running drizzle-kit push against $ENVIRONMENT database"
          npx drizzle-kit push
          echo "drizzle-kit push completed successfully"

      - name: Migration summary
        if: always()
        env:
          ENVIRONMENT: ${{ inputs.environment }}
          FIREBASE_PROJECT_ID: ${{ inputs.firebase_project_id }}
        run: |
          {
            echo "## DB Migration"
            echo ""
            echo "| Field | Value |"
            echo "|-------|-------|"
            echo "| Environment | \`$ENVIRONMENT\` |"
            echo "| Firebase Project | \`$FIREBASE_PROJECT_ID\` |"
            echo "| Status | $([[ '${{ job.status }}' == 'success' ]] && echo 'SUCCESS' || echo 'FAILED') |"
          } >> "$GITHUB_STEP_SUMMARY"
```

The `--url` flag is documented as an override — both approaches work. Using the env var is cleaner and consistent with local dev behavior.

## Risks & Pitfalls

### Pitfall 1: apphosting.prod.yaml is not a partial diff — it's a complete config

**What goes wrong:** Developer creates `apphosting.prod.yaml` with only the DATABASE_URL override. FAH uses `--config apphosting.prod.yaml` exclusively — it does NOT merge with `apphosting.yaml`. The prod build fails because `NODE_AUTH_TOKEN` (GITHUB_PACKAGES_TOKEN) is missing, causing `npm ci` to 401 against GitHub Packages.

**Prevention:** Both `apphosting.yaml` and `apphosting.prod.yaml` must include the full env block including `NODE_AUTH_TOKEN: secret: GITHUB_PACKAGES_TOKEN` with `availability: [BUILD]`. The draft overlays above include this.

**Warning sign:** Prod deploy shows `npm error code E401` or `npm error 401 Unauthorized` in Firebase build logs.

### Pitfall 2: shared-workflows is a SEPARATE REPO — executor must clone it

**What goes wrong:** Executor tries to edit shared-workflows files from the admin repo worktree. The files live in `MyAlterLego/shared-workflows` — a completely different git repository at `~/claude/MyAlterLego/shared-workflows/`.

**Prevention:** Plans 07.5-02 and 07.5-03 must explicitly direct the executor to `cd ~/claude/MyAlterLego/shared-workflows/` and work from that directory. Commits go to that repo's branch, not to admin.

**Path verified:** `/Users/mikegeehan/claude/MyAlterLego/shared-workflows/.github/workflows/` — confirmed the directory exists and the live files were read from there.

### Pitfall 3: Tag v4 AFTER admin canary, not before

**What goes wrong:** If v4 is tagged before canary verification and it has a bug, any consumer doing `@v4` gets broken deploys. Phase 2-03 had this exact issue (canary exposed a version extraction bug after v2 was tagged — fortunately v2 had no other consumers yet).

**Prevention:** Plan 07.5-02 writes the v4 files and gets them merged. Plan 07.5-03 is the ONLY plan that pushes the `v4` tag, and only after admin canary deploy is confirmed live.

### Pitfall 4: NEXTAUTH_URL in apphosting.yaml must match the actual admin-dev URL

**What goes wrong:** The draft `apphosting.yaml` sets `NEXTAUTH_URL = https://admin-dev.triarch.dev` as a placeholder. The actual FAH default URL for the `<app>-dev` backend is the hosted.app pattern: something like `admin-dev.<region>.hosted.app`. If NEXTAUTH_URL doesn't match, Google OAuth redirect will be rejected.

**Prevention:** Plan 07.5-03 (post-backend-creation) must update `apphosting.yaml` with the real FAH URL for the admin-dev backend before running the canary deploy. Mike records the real URL when he creates the backend (it's shown in the Firebase Console and RUNBOOK.html captures it).

**Action required in plan:** After HUMAN creates admin-dev backend (RUNBOOK.html step), executor reads the FAH default URL and updates `apphosting.yaml` NEXTAUTH_URL before pushing the dev branch canary.

### Pitfall 5: DATABASE_URL_DEV secret must be set BEFORE first deploy to admin-dev

**What goes wrong:** Dev deploy lands on admin-dev backend before `DATABASE_URL_DEV` secret is set. The app starts but crashes on any DB operation with a missing env var error.

**Prevention:** RUNBOOK.html step P7.5-xx covers this. Plan 07.5-03 must assert the secret is set as a prerequisite before triggering the canary deploy. Verify with: `firebase apphosting:secrets:access DATABASE_URL_DEV --project triarch-dev-website` (should return a connection string).

### Pitfall 6: drizzle-kit push needs drizzle.config.ts in consumer repo root

**What goes wrong:** db-migrate.yml checks out the consumer repo and runs `npx drizzle-kit push`. If `drizzle.config.ts` is absent (e.g., www or portal don't have Drizzle configured), drizzle-kit fails with a config not found error.

**Prevention:** db-migrate.yml should be called only from repos that use Drizzle ORM. For v2.0, that means admin and portal. www and other static sites should not call this workflow. Document in Step 11 of onboarding-projects.md.

### Pitfall 7: `firebase apphosting:secrets:access` outputs to stdout with potential trailing newline

**What goes wrong:** The `read_secret` step captures `firebase apphosting:secrets:access` output. If the secret value has a trailing newline (common for multi-line secrets like `GITHUB_APP_PRIVATE_KEY`), the `database_url` output will have a trailing `\n`, which is invalid in a CRDB connection string and will cause drizzle-kit to fail.

**Prevention:** In db-migrate.yml, trim the output: `DB_URL=$(firebase apphosting:secrets:access "$SECRET_NAME" --project "$FIREBASE_PROJECT_ID" | tr -d '\n')`. DATABASE_URL is a single-line URI, so `tr -d '\n'` is safe.

### Pitfall 8: v3 tag must not move when v4 is created

**What goes wrong:** If the git tagging command in Plan 07.5-03 accidentally moves the `v3` tag (e.g., `git tag -f v3` or `git tag v4 v3^{}` running in the wrong order), consumers still on v3 (CRM, portal, etc.) will break.

**Prevention:** Tag creation command must be: `git tag v4 <specific-commit-sha>`. Verify v3 is still pointing at the correct commit after v4 is pushed: `git tag -l --sort=-version:refname | head -5`.

### Pitfall 9: admin canary ci-cd.yml must pass `app_hosting_backend` explicitly

**What goes wrong:** The v4 `deploy-firebase.yml` derives the dev backend name by appending `-dev` to the `app_hosting_backend` input. If admin's `ci-cd.yml` does not pass `app_hosting_backend` (relying on the `firebase_project_id` fallback), the derived dev backend name will be `triarch-dev-website-dev` (wrong) instead of `admin-dev` (correct).

**Prevention:** When admin bumps to v4, explicitly set `app_hosting_backend: admin` in the workflow call — the v4 logic then derives `admin-dev` for env=dev. Alternatively, the v4 workflow can accept a `dev_backend_name` override input. Recommend explicit `app_hosting_backend: admin`.
