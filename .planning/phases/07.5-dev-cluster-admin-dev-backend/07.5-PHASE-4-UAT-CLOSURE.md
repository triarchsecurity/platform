## Scenario B — Rebase Conflict (SC-2)

**Setup:** Create a test branch `test/uat-conflict` from `dev`. Make a change that conflicts with a recent main commit (e.g. modify the same line of a source file that was changed on main since the branch was cut). Push `test/uat-conflict`.

Detailed steps from RUNBOOK D-2:
1. From admin `main`: edit a line in any source file, commit, push to a new branch `feat/uat-base-A` and approve to merge it (creates the baseline change on main)
2. Branch `test/uat-conflict` from `main` BEFORE the step 1 merge lands → edit the SAME line differently, commit, push
3. Approve `test/uat-conflict` via admin UI

**Action:** Approve `test/uat-conflict` from the customer release page — dispatches `promote-branch.yml@v3`.

## Scenario D — Concurrency (SC-4)

**Setup:** Create TWO test branches from `dev` simultaneously: `test/uat-concurrent-A` and `test/uat-concurrent-B`. Each touches a DIFFERENT file (no cross-conflict). Push both. Let CI run on both.

**Action:** Approve `test/uat-concurrent-A` first via admin UI; while A's promote-branch.yml is still running, approve `test/uat-concurrent-B`. Both `promote-branch.yml@v3` runs should kick off. The first to merge updates main; the second auto-rebases on the new main and re-runs CI.
