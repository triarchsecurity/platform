## Implementation Decisions

### Dev cluster provisioning (success criterion 1)

- **D-01:** Provision a NEW dedicated CockroachDB Cloud cluster — separate physical/billing entity from the existing prod cluster (`triarchdev-24092`). Naming follows Cockroach Cloud convention (cluster name appears in connection strings); recommend `triarchdev-dev` or accept Cockroach-generated suffix.
- **D-02:** Region — same as prod (typically `us-east1`) for parity. Latency is irrelevant for dev; matters for migration-test fidelity.
- **D-03:** Sizing — smallest CRDB Cloud tier (`Basic` 2 vCPU). Dev workload is low; can scale up if needed.
- **D-04:** Cluster creation is a HUMAN action (Mike provisions via Cockroach Cloud console). Plan emits a HUMAN-UAT step with the exact provisioning checklist + acceptance criteria.

### Per-project dev DB naming + creation (success criterion 1)

- **D-05:** Naming convention `<app>_dev`: `admin_dev`, `portal_dev`, `darksouls_dev`, `tmi_dev`, `truthtreason_dev`, `www_dev` (six DBs). Matches success criterion verbatim.
- **D-06:** Creation method — single one-shot SQL script `scripts/provision-dev-dbs.sql` containing `CREATE DATABASE` for each of the six. Mike applies it post-cluster-provision via `cockroach sql --url <new-cluster-url> -f scripts/provision-dev-dbs.sql`.
- **D-07:** Owner role — default admin role on the new cluster (no per-DB role differentiation in v2.0; per-DB roles are a v3 deferral).
- **D-08:** Schema seeding — empty databases. First `drizzle-kit push` runs through the new `db-migrate.yml` workflow against `<app>_dev`'s connection URL. Initial seed data per-project is OUT OF SCOPE (Mike seeds manually if a UAT scenario needs sample data).

### Per-app dev backend (FAH) provisioning (success criterion 2, 4)

- **D-09:** Firebase project — SAME project as prod for Triarch-owned apps (admin, portal, www). Customer-owned apps (truthtreason, darksouls, tmi) get `<app>-dev` backends in the existing customer GCP project. NO new GCP projects.
- **D-10:** Git branch convention — `dev` (single canonical name across all repos). FAH `<app>-dev` backend is configured to deploy on push to `dev`.
- **D-11:** DATABASE_URL_DEV secret — set per-backend via `firebase apphosting:secrets:set DATABASE_URL_DEV` after the dev cluster is up. Each `<app>-dev` backend's `apphosting.yaml` references `DATABASE_URL_DEV`. Each `<app>` (prod) backend's `apphosting.prod.yaml` continues to reference `DATABASE_URL`.
- **D-12:** Domain — FAH default URL pattern `<app>-dev.<region>.hosted.app`. NO custom DNS for dev (avoid DNS sprawl + cert management).
- **D-13:** Auth on dev backends — same NextAuth + Google OAuth allowlist as prod (no auth-bypass). Real-auth flow tested in dev. Customer-admin allowlist may differ per-project (handled per-app config).
- **D-14:** Backend creation is a HUMAN action — `firebase apphosting:backends:create <app>-dev --git-branch=dev` per project. Plan emits a runbook step.

### apphosting.yaml overlay convention (success criterion 4)

- **D-15:** Convention — `apphosting.yaml` = base config (dev defaults including `DATABASE_URL_DEV` and any dev-specific env), `apphosting.prod.yaml` = prod overrides (`DATABASE_URL` and prod-only env). Matches the truth+treason pattern.
- **D-16:** Selection mechanism — `deploy-firebase.yml@v4` passes `--config apphosting.prod.yaml` to `firebase apphosting:rollouts:create` when `environment=prod`; uses default `apphosting.yaml` when `environment=dev` or unset. FAH respects the `--config` flag for per-rollout config selection.
- **D-17:** Documentation — `docs/onboarding-projects.md` Step 11 (NEW) documents the convention with exact YAML examples for both files.

### shared-workflows v4 design (success criterion 3)

- **D-18:** New `environment: dev|prod` input on `deploy-firebase.yml`. Default `prod` for backwards compatibility (consumers not yet migrated continue working). When `environment=dev`: targets `<app>-dev` backend, uses `DATABASE_URL_DEV`, defaults git ref to `dev`. When `environment=prod`: targets `<app>` backend, uses `DATABASE_URL`, defaults git ref to `main`.
- **D-19:** New reusable `db-migrate.yml` workflow:
  - Inputs: `environment` (dev|prod), `firebase_project_id`, `consumer_repo`
  - Steps: checkout consumer repo at the env's branch → `npm ci` → `firebase apphosting:secrets:access DATABASE_URL_DEV --project <fbid>` (or DATABASE_URL for prod) → `npx drizzle-kit push --url=$DATABASE_URL`
  - Replaces the manual `firebase apphosting:secrets:access | psql -f` ritual that's currently in onboarding-projects.md
- **D-20:** Tag v4 on `MyAlterLego/shared-workflows`. v3 (Phase 4) stays for consumers not yet migrated. Migration order: tag v4 → admin canary → verify 1 dev deploy succeeds → bump CRM and other consumers in subsequent passes.
- **D-21:** Backwards compat — consumers passing no `environment` input get `prod` behavior identical to v3. Consumers can opt into dev by adding `environment: dev` to their `ci-cd.yml` workflow_call invocation.

<specifics>
## Specific Ideas

- **`dev` branch convention** matches what the admin repo already does for feature branches (`feat/...`); dev is just one named branch sitting alongside main.
- **db-migrate.yml** should be invoked manually via workflow_dispatch initially; later phases may auto-trigger on schema.ts diff in the dev branch. v2.0 keeps it manual to avoid surprise drops.
- **One-shot SQL** for cluster provisioning is preferable to per-DB scripts — single file `scripts/provision-dev-dbs.sql` with all six `CREATE DATABASE` statements is easier to audit + re-run idempotently (`CREATE DATABASE IF NOT EXISTS`).
- **Phase 4 UAT closure** is a verification deliverable, not new code. Document exact run-throughs + expected results in `07.5-PHASE-4-UAT-CLOSURE.md`. Frame as a runbook, not a test suite.
- **Backwards compat** is paramount for shared-workflows — every consumer NOT yet migrated to v4 must keep working. The default `environment: prod` ensures this.
- **Cost** — adding a dev cluster + 6 dev backends increases monthly Firebase + CRDB cost. Smallest tier on each side keeps it ~$50-100/mo extra. Document in onboarding-projects.md so customer-owned projects know the marginal cost.

</specifics>
