# Phase 23.1: Portal UI Polish - Context

**Gathered:** 2026-05-09
**Status:** Ready for planning
**Trigger:** First customer sign-in to portal v0.4.6 surfaced 4 navigability + clarity gaps Mike reported with screenshots.

<domain>
## Phase Boundary

Close the navigability + status-clarity gaps reported on portal v0.4.6 first sign-in. Four scoped UI changes:

1. **Sub-nav** linking Releases / Bugs / Features on the project surface
2. **Status column rewrite** rendering real `releaseLogs.status` with color-coded badges + "needs your review" emphasis
3. **Empty-state copy** replacing cryptic `PROD --` with "Not yet released"
4. **Staff "preview as customer" toggle** for end-to-end testing without account switching

Out of scope: rendering changes to bug/feature detail pages, additional new routes, schema changes, server logic beyond the toggle's cookie read.

</domain>

<decisions>
## Implementation Decisions

### UX-01: Sub-nav layout (16 of 16 decisions captured via discuss-phase)
- **Placement:** Horizontal tabs **below the project header**, above page content. Single shared component used by all `/projects/[slug]/*` routes (Releases, Bugs, Features). Same UI applies to every project — no per-project conditional logic.
- **Active styling:** Underline + color-shifted label, matching admin's existing tab pattern (release-logs filter chips). NOT a filled pill.
- **Tab list:** Three fixed tabs in order: **Releases / Bugs / Features**. Project home / settings tabs deferred (no settings page exists yet).
- **Mobile breakpoint:** Horizontal scroll if cramped (≤ 320px). NOT a hamburger menu — only 3 items.

### UX-02: Status column rewrite
- **Where:** Replace the current `STATUS` column (which incorrectly renders `env`). Add a separate `ENV` column for env, so both pieces of info are visible.
- **Status values to render:** `pending_approval`, `approved`, `rejected`, `promoted`, `superseded`, `dev` (matches schema enum). NULL/unset → render "dev" as default.
- **Color scheme:** Saturated accent palette matching www.triarch.dev dashboard:
  - `pending_approval` → **amber** (saturated, e.g. amber-400 on amber-500/20)
  - `approved` → **teal/cyan** (saturated, awaiting deploy)
  - `rejected` → **pink/red** (saturated)
  - `promoted` → **emerald-green** (saturated, "in prod")
  - `superseded` → **zinc-grey** (muted, "history")
  - `dev` → **zinc-blue** (muted, "intermediate")
  - Validate against existing `StatusPill.tsx` colors from Phase 23-01 — they should already align with admin's release-logs pills which already match www.triarch.dev's palette. Adjust any drift.
- **"Needs your review" emphasis:**
  - Subtle row highlight on `pending_approval` rows: amber 5% bg + amber left border
  - Branch-section header gets a count badge: e.g. "**main** (2 pending review)" when ≥1 row in section is pending_approval
- **Sort/filter:**
  - Default order unchanged: newest first
  - Add an optional client-side filter chip: "Pending review only" — toggleable, off by default

### UX-03: Empty-state copy on project cards
- **Prod row when no prod release exists:** Render `PROD: Not yet released` (no version number, no timestamp). Single line, same row layout as populated cards.
- **Dev row when no dev release exists:** Same — `DEV: Not yet released`. Consistent with prod.
- **Timestamp rendering:** Hide the "X ago" timestamp entirely when there's no row to relate it to. NOT "—", NOT "never".
- **Card visual when both rows empty:** Standard card style. No "inactive project" greyed treatment — projects can go live at any time; greying might confuse.

### UX-04: Staff "preview as customer" toggle
- **Toggle location:** Affordance in the **existing staff banner** ("You are signed in as Triarch staff. Customer features are read-only here..."). Add a "**Preview as customer →**" link on the right (mirroring the "Switch to admin.triarch.dev" link layout).
- **Storage:** Cookie `portal-preview-as-customer=1`, 1-hour TTL, host-only (`__Host-` prefix in production), no JS access. Auto-expires — staff doesn't get permanently stuck in customer view.
- **Scope of preview:**
  - Suppress the staff banner
  - Render customer action UI as if `isStaff=false` (approve/reject/feedback/branch-swap buttons visible AND clickable)
  - User context's `isStaff` flag remains true server-side; only UI suppression toggles
  - DO NOT change anything else (no admin-only sections to hide on portal — portal is customer-only by design)
- **Exit affordance:**
  - Persistent fixed banner at top of every page: "👁 Preview mode active (Xm remaining) — Exit preview"
  - Click "Exit preview" → clears cookie + reloads page → returns to staff view
  - Banner styled subtle but always visible (small, top-edge, eye icon)

### Cross-cutting (Claude's Discretion)
- All 4 changes are dev-portal repo only — no admin code changes
- Mobile-responsive across all changes (Phase 21 PORTAL-04 standing rule)
- Vitest tests per change; existing test scaffolding in portal works
- Status pill color audit can extract a shared color map to `@triarchsecurity/triarch-shared` if helpful, OR keep portal-local; pick based on whether admin's existing pills are already in the shared package
- Cookie name follows portal's NextAuth `__Host-` convention

</decisions>
