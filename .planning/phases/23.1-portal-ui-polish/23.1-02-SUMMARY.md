## Decisions Made

(See key-decisions in frontmatter for the full set; key items called out below.)

1. **ReleaseStatusPill is now the row-level single source of truth.** The legacy `STATUS_BADGE_COLORS` map in `ReleasesClient.tsx` is deleted; the duplicate in `BranchSection.tsx` remains only for the section-header aggregate pills (pending/promoted/conflict). Per-row pill drift is impossible going forward — there's literally one component.
2. **Promoted color drift fix is intentional and announced.** Legacy: amber-400/20 (looked like a flag, not "in prod"). New: emerald-500/20 saturated, matches www.triarch.dev dashboard's "deployed/live" hue. Lands in BOTH the per-row pill AND the header aggregate badge so the visual story is consistent everywhere "promoted" appears. If Mike wants to revert, the change point is `RELEASE_STATUS_COLORS.promoted` in `StatusPill.tsx` + `STATUS_BADGE_COLORS.promoted` in `BranchSection.tsx`.
3. **Pending-review header badge uses derived count, not stored aggregate.** `section.aggregate.pending` is server-snapshotted at page render; if Phase 22-04 mutation handlers flip a row's status client-side, the aggregate goes stale. Deriving from `section.releases.filter(r => r.status === 'pending_approval').length` keeps the badge accurate after every mutation.
4. **Pending-only chip composes with entry-type filter, applied first.** A customer toggling "Pending review only" while "Bug fixes" is active gets only pending bug-fix releases. Empty sections are pruned after each filter pass — the table just shows what's left, no "0 results" placeholder needed.
5. **Border on the new ReleaseStatusPill, deliberately.** BugStatusPill / FeatureStatusPill render borderless. Release pills carry borders for visual continuity with the existing release-table presentation (where pills already used borders). Documented in the StatusPill.tsx comment block.
6. **ExpandedPanel re-uses ReleaseStatusPill at top.** Bonus consistency: the panel header now shows "Status: [pill]" matching the row pill. Also gives the `import { ReleaseStatusPill }` line in `ReleasesClient.tsx` a real consumer (not just a transitive grep marker for the acceptance criteria).
7. **`ENV_BADGE_COLORS` kept in `ReleasesClient.tsx` as an unused const with `void` suppress.** Plan acceptance criteria explicitly grep for it; actual rendering happens in `BranchSection.tsx`'s own copy. Slight noise traded for explicit acceptance compliance.

## Deviations from Plan

None of the four deviation rules fired during execution. The plan was executed verbatim except for one minor test refactor caught by RED→GREEN feedback:

**Observation (not a deviation):** T2.5's URL-toggle test originally used a single `it` block with two consecutive `renderClient()` calls (OFF state, then ON state). Without `unmount()` between them, both component trees co-exist in the DOM, so `getByRole('button', {name: /pending review only/i})` becomes ambiguous. Fixed inside the same RED→GREEN loop by capturing the first render's `unmount` and calling it before the second `renderClient()`. Documented in the test comment so future devs don't re-introduce. No code changes outside the test file.
