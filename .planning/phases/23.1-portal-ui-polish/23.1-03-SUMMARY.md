## Decisions Made

(See key-decisions in frontmatter for the full set; key items called out below.)

1. **Empty-state row uses `flex items-baseline gap-2` (no `justify-between`).** A populated row reads "Prod | v1.2.0 | 2 hr ago" with three columns; the empty row reads "Prod: | Not yet released" with two tokens hugged left. Visually distinct enough that the reader registers "different state" but keeps row-height parity (no layout jump between populated and empty).
2. **Source text is `Prod:` / `Dev:`; rendered surface is `PROD:` / `DEV:` via existing CSS.** The label span already had `text-xs uppercase tracking-wide` from the populated path; reusing it keeps visual continuity. Tests assert on the DOM text (`Prod:`) since CSS doesn't transform DOM-string content. CONTEXT.md UX-03 D-01's `PROD: Not yet released` literal is what the user sees rendered — exactly correct.
3. **Test pattern: `renderToStaticMarkup` directly on the awaited server component.** Same approach Phase 21 used for `releases/page.test.tsx` (membership 404 enforcement). Mocks: `getPortalSession` + `getCurrentUserContext` for auth, `getProjectPipelineSummaries` as the unit-under-test boundary, `db.select` chain returns project rows. No jsdom — pure string-substring assertions on serialised HTML.
4. **Timestamp hidden — not rendered as "—" or "never".** Per CONTEXT.md UX-03 D-03: hide the span entirely. Confirmed in T1.5 by counting "ago"/"just now" occurrences in the rendered HTML — exactly one for a no-prod-has-dev tile (the dev side only).
5. **T1.7 defense-in-depth (missing summary entry).** A project key with a member row but no `release_logs` rows whatsoever produces a `summaryByKey[key] === undefined` lookup. The existing `summary?.prodVersion` optional-chaining returns `undefined`, falling to the empty-state branch — same render output as a summary with all-null fields. Test confirms.
