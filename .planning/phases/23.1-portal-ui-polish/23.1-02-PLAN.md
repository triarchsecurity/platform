<objective>
Replace ReleasesClient's incorrect status column (which currently renders `env` as status, per CONTEXT.md UX-02) with the real `releaseLogs.status` enum + separate ENV column + pending-approval emphasis affordances.

Purpose: Customers reported on first sign-in that the "STATUS" column shows "dev / prod" — that is env, not lifecycle status. The actual `releaseLogs.status` (pending_approval, approved, rejected, promoted, superseded) is the customer-actionable signal: a customer admin needs to see at a glance "what needs my review?" not "what env is this?". This plan separates the two and emphasizes pending-approval rows so the customer surface is self-evident.

Output:
- New `ReleaseStatusPill` export in StatusPill.tsx with the 6-status saturated color palette from CONTEXT.md UX-02 (amber / teal / red / emerald / zinc-grey / zinc-blue), validated against admin's release-logs page palette.
- ReleasesClient.tsx: new STATUS column rendering `ReleaseStatusPill` from `row.status` (or 'dev' fallback for null); ENV column kept separately; pending-approval row highlight (amber-500/5 bg + border-l-2 border-amber-500).
- BranchSection.tsx header: count badge "N pending review" amber pill when section has ≥1 pending_approval row.
- New "Pending review only" filter chip in ReleasesClient (URL-mirrored `?pending=1`); off by default; toggle on to filter table.
- Vitest tests per status (color + display text), per row highlight, per section badge, per filter chip URL state.
- Portal patch bump v0.4.7 → v0.4.8.
</objective>
