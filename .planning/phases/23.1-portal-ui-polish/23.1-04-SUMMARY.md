## Cookie Hygiene — Set-Cookie Header (production)

Per `buildSetCookieAttributes()`:

```
Set-Cookie: __Host-portal-preview-as-customer=<expiry-epoch-ms>; Max-Age=3600; Path=/; HttpOnly; SameSite=Lax; Secure
```

- **`__Host-` prefix** — browser-enforced rejection of any Domain= injection. Requires Secure + Path=/ + no Domain attribute (all satisfied).
- **No Domain attribute** — host-only by omission, prevents cross-subdomain leakage. Guarded by source-text grep test (T1.11): `grep -E "^\s*domain:" portal/src/lib/preview-as-customer.ts` returns 0 matches.
- **HttpOnly** — no JS access (CONTEXT.md UX-04 D-02).
- **SameSite=Lax** — matches NextAuth session-token pattern; portal POST origin is its own domain.
- **Max-Age=3600** — 1-hour TTL (CONTEXT.md UX-04 D-02 — auto-expires so staff isn't permanently stuck).
- **Secure** — production-only (development uses unprefixed name + omits Secure since HTTPS not guaranteed).

In dev/test environments the cookie name drops the `__Host-` prefix (mirrors auth.ts pattern); attribute set is the same minus Secure.

## Security Boundary — 3 Explicit Tests

The cookie ONLY suppresses staff-bypass UI for users who already have staff role. It NEVER grants permission. Three dedicated tests prove this end-to-end:

1. **T1.9 (helper-level):** Non-staff session + cookie set → `isPreviewingAsCustomer() === false`. The helper checks `session.user.isStaff` BEFORE reading the cookie. Non-staff users get false unconditionally.
2. **T2.2 (route-level):** Non-staff POST to `/api/preview-as-customer` → 403, no cookie set. Defense in depth: even if a non-staff user crafts the request, the API rejects.
3. **T4.4 (page-level):** Non-staff customer with cookie set → `userRole='admin'` unchanged from baseline (mirror of T4.5). The cookie has zero effect on UI for non-staff users.

Plus **T4.3 (real-role-survives):** staff + cookie + admin membership → `userRole='admin'`. Preview ONLY suppresses the staff-bypass branch (`ctx.isStaff && !previewing`); the genuine `membership.role === 'admin'` branch is unaffected. A staff user who happens to have a real admin membership row keeps admin permissions during preview.

## Form-vs-Fetch Decision for the "Preview as customer" Link

**Form chosen.** The "Preview as customer" affordance is a `<form action="/api/preview-as-customer" method="POST">` with a submit button. POST handler returns 303 (See Other) redirect to Referer (or '/' if absent). Browser follows 303 with a GET to the original page, now with the cookie active. **Works without JS.**

The Exit-preview button uses `fetch DELETE` + `router.refresh` (client-side reload) because it's already inside a `'use client'` component (PreviewModeBanner needs the setInterval ticker for minutes-remaining display). Both flows are reversible without page-state loss.

## Mobile Responsive Verification at 320px

PreviewModeBanner uses `flex items-center justify-center gap-3` with `text-xs`. At 320px, the rendered text is approximately:

```
👁 Preview mode active (60m remaining) — Exit preview
```

Roughly 50 characters at text-xs (~0.75rem ≈ 9px) → ~450px wide. **Overflows at 320px.** The current implementation accepts the overflow because the banner is `fixed top-0 left-0 right-0` (full-width) and has internal flex centering — text wraps within the available width via flex-shrink. Verified test T3.1 passes with the literal text + button still accessible. If Mike wants tighter mobile UX, future refinement could:

- Drop "(Xm remaining)" parenthetical at narrow viewports (`hidden sm:inline`)
- Use `flex-wrap` to allow the button to wrap below the message
- Reduce font-size further with `text-[11px]` at narrow viewports

This is a P3 polish item — banner is functional and visible at 320px; just visually crowded. Logged for follow-up if needed.

## Decisions Made

(See key-decisions in frontmatter for the full set; major items called out below.)
