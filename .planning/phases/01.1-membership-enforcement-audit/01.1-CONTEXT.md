# Phase 1.1: Membership Enforcement Audit - Context

**Gathered:** 2026-05-03
**Status:** Ready for planning
**Source:** Direct write (decisions already articulated in REQUIREMENTS.md MEMBER-AUDIT-01..10 and ROADMAP.md Phase 1.1 success criteria; this is a focused gap-closure phase, no open questions)

<domain>
## Phase Boundary

Audit and correct access control on every existing admin/API endpoint that currently uses `requireAdmin()` from `src/lib/api-auth.ts`. Phase 1's `auth.ts` cutover legitimately allows project-member emails to sign in, but `requireAdmin()` only checks for a session — so signed-in non-staff customer admins now bypass intended access boundaries on 32 endpoints. This phase classifies each endpoint, applies the correct check (staff-only / membership-aware list / membership check on detail), and verifies with the live `mike@mikegeehan.com` test account that data and destructive operations are correctly scoped.

</domain>

<decisions>
## Implementation Decisions

### Helpers (added to `src/lib/api-auth.ts`)
- **Rename, don't break**: `requireAdmin` → `requireSignedIn`. Keep a one-line `export const requireAdmin = requireSignedIn` alias for any in-flight branch; mark `// deprecated — use requireSignedIn or requireStaff per endpoint policy. remove in v1.15` and surface in the v1.15 backlog. We do not chase down 32 callsites for a rename in the same PR that's also changing access logic.
- **`requireStaff()`** consults `getCurrentUserContext(session)` from Phase 1's auth-context. Returns `{error: 401, ...}` if unauthenticated, `{error: 403, ...}` if `!ctx.isStaff`, otherwise `{error: null, session, ctx}`.
- **`requireMembership(projectKey)`** consults `getCurrentUserContext(session)`. Staff bypass: `ctx.isStaff` returns `{error: null, ctx}` for ALL projects. Non-staff: returns 403 if `!ctx.memberships.some(m => m.project_key === projectKey)`. The 403 is for callers that already know the project exists — pages should mirror Phase 1's "404 not 403 to avoid project-existence leak" pattern.

### Endpoint classification policy

For each of the 32 endpoints currently using `requireAdmin()`, classify into:

| Class | Definition | Action |
|-------|-----------|--------|
| `staff-only` | Operates on platform infrastructure or cross-project state. Customers must NEVER be able to call. | Replace `requireAdmin()` with `requireStaff()`. |
| `project-list` | Returns a collection of project-scoped rows (release-logs, bug-reports, feature-requests). | Use `requireSignedIn()` + filter results: staff get unfiltered; non-staff get `WHERE project IN (ctx.memberships.map(m => m.project_key))`. Empty memberships → empty result (not 403). |
| `project-detail` | Operates on a single project's row, identified by URL param or body. | Use `requireSignedIn()` + `requireMembership(row.project)` after the row is fetched. Non-member non-staff → 404 (mirror page-level no-leak pattern). |
| `unclear` | Not obviously project-scoped or platform-wide on first read. | Read the endpoint body, decide. Default to `requireStaff` if customer access has no plausible benefit. |

### Specific classifications (initial — refine in PLAN)

**staff-only (must reject non-staff with 403):**
- `/api/platform/projects/[id]` (PUT, DELETE) — edit/remove project metadata
- `/api/platform/projects/[id]/destroy` — cascading project decommission
- `/api/platform/projects/scaffold-repo`, `/provision-db`, `/provision-dns`, `/sync-state`
- `/api/platform/projects/tools` (and `/tools/scaffold`)
- `/api/platform/navigation/sections`, `/sections/[id]`, `/pages`, `/pages/[id]`, `/subpages`, `/subpages/[id]`, `/reorder`, `/admin`
- `/api/platform/settings`
- `/api/platform/access-logs`
- `/api/platform/release-logs/backfill`
- `/api/platform/webhooks/backfill`
- `/api/platform/report-section-types` (verify in PLAN — likely platform-wide)
- `/api/platform/service-offerings`, `/service-offerings/[id]` (verify in PLAN — possibly project-scoped, possibly platform)

**project-list (filter for non-staff):**
- `GET /api/platform/release-logs`
- `GET /api/platform/bug-reports`
- `GET /api/platform/feature-requests`

**project-detail (membership check on `row.project`):**
- `GET/PUT /api/platform/release-logs/[id]`
- `GET/PUT /api/platform/bug-reports/[id]`
- `GET/PUT /api/platform/feature-requests/[id]`

### POST handlers on list endpoints

Where the same route file exposes POST (e.g., creating a bug report from the admin UI), apply membership check on the `project` field of the request body for non-staff. Staff can post for any project. This handles both the "as a member I file a bug for my own project" case and the "as staff I'm filling in retroactively" case.

### Page-level checks

Server-component pages under `src/app/admin/modules/` (release-logs viewer, bug-reports viewer, feature-requests viewer) currently use client-side `fetch` to the API endpoints. With the API endpoints now membership-aware, the pages inherit the filter automatically. Verify this assumption per page in the PLAN; add inline guards only if a page reads the DB directly via server components.

### Verification approach

After all classifications applied:
1. Manual API curl with mike@mikegeehan.com session cookie — confirm 403 on staff-only, filtered results on lists, 404 on cross-project detail.
2. Manual sign-in via browser — confirm UI matches API behavior (release-logs page shows only darksouls, bug-reports same, etc.).
3. Sanity check with mike@triarchsecurity.com — staff experience unchanged.

### Claude's Discretion
- Order of application across endpoints (suggested: helpers first, then staff-only batch, then list batch, then detail batch — review in PLAN)
- Naming of internal type for the auth-context return shape (already exists as `UserContext` from Phase 1)
- Whether to write a single shared "withMembership" wrapper for the list endpoints' filter logic, or inline the WHERE clause each time — implementer chooses based on call-site count

## Existing Code Insights

### Reusable Assets
- `src/lib/auth-context.ts` exports `getCurrentUserContext(session): UserContext | null` from Phase 1 — single source of truth for `{email, isStaff, memberships[]}`
- `src/lib/api-auth.ts` has the existing `requireAdmin` to rename + the new helpers will live alongside
- Phase 1's manage-members route handlers (`src/app/api/platform/projects/[id]/members/...`) already demonstrate the `requireStaff` pattern locally — this phase generalizes to `src/lib/api-auth.ts`
- Phase 1 Plan 01-04 (`/api/platform/projects/route.ts` membership filter) demonstrates the project-list pattern

### Established Patterns
- Drizzle queries via `db.select().from(table)` + `inArray(col, values)` for IN-list filtering
- 401 / 403 / 404 conventions: 401 unauthenticated, 403 forbidden (staff-only refused for non-staff), 404 not-found-or-not-a-member (no info leak when project membership fails)
- All route handlers `await getServerSession(authOptions)` first, then check via helper

### Integration Points
- Every endpoint in `src/app/api/platform/` that imports `requireAdmin` from `@/lib/api-auth` (32 files) — all need updating
- No changes needed to `src/lib/auth.ts` itself; the `signIn` callback already correctly distinguishes staff vs non-staff via `getCurrentUserContext`
- Server-component admin pages may not need code changes if they delegate to the API; verify per page in PLAN

<specifics>
## Specific Ideas

- Helpers signature suggestion (final shape TBD by implementer):
  - `async function requireSignedIn(): Promise<{error: NextResponse | null, session: Session | null}>` — same as today's requireAdmin
  - `async function requireStaff(): Promise<{error: NextResponse | null, session: Session | null, ctx: UserContext | null}>` — extends with ctx, returns 403 for non-staff
  - `async function requireMembership(projectKey: string): Promise<{error: NextResponse | null, session: Session | null, ctx: UserContext | null}>` — staff bypass; otherwise 403 if no membership
- For the project-list filter, a small helper like `function projectFilterFor(ctx: UserContext): SQL | null` returning `null` for staff (no filter) and `inArray(table.project, ctx.memberships.map(...))` for members — keeps the pattern uniform across release-logs, bug-reports, feature-requests
- POST endpoints on project-scoped routes: validate `body.project` is in `ctx.memberships` for non-staff before allowing the insert
- DETAIL endpoints (`/[id]`): fetch the row first, THEN check membership on `row.project`. Returning 404 (not 403) prevents leaking "this project exists, you just can't see it" info

</specifics>
