# Phase 01.1 Plan 04: Project-List Membership Enforcement Summary

**One-liner:** GET membership filter via `inArray(table.project, projectKeys)` + POST `body.project` membership check across 5 project-list endpoints, with staff bypass on all paths.

## What Was Built

Five project-scoped LIST endpoints migrated from `requireAdmin` to membership-aware access control. The structural pattern is identical across all five files:

**GET pattern:**
1. `requireSignedIn()` — any authenticated session passes
2. `getCurrentUserContext(session)` — fetch membership data
3. If `ctx && !ctx.isStaff`: compute `projectKeys` from `ctx.memberships` (filtering out `'*'` wildcards)
4. If `projectKeys.length === 0`: return `200 + { KEY: [], total: 0, ... }` (NOT 403)
5. Otherwise: `conditions.push(inArray(TABLE.project, projectKeys))` — appended after existing query-param filters
6. Staff and DB-error fallback (`ctx === null`) see unfiltered results

**POST pattern:**
1. `requireSignedIn()` — any authenticated session passes
2. Parse body, run existing validation (400 on missing fields — BEFORE the membership check)
3. `getCurrentUserContext(session)` — fetch membership data
4. If `ctx && !ctx.isStaff`: `ctx.memberships.some(m => m.project_key === body.project)` — if false → 403
5. Staff and DB-error fallback let the insert through
