## Special Cases Documented

### reports/[id]/route.ts — DELETE caution
Classified `project-detail` but DELETE should be restricted to staff-only within Plan 05. A customer member should not be able to delete a report deliverable. Plan 05 implementer must branch on method: GET/PUT use membership check, DELETE uses requireStaff.

### projects/route.ts — dual GET/POST behavior
GET already implements the membership filter from Plan 01-04 legacy work. Plan 04 need only swap the `requireAdmin` import to `requireSignedIn` and leave the filter logic intact. POST must be wrapped with `requireStaff()` — project creation has no customer use case.

### access-logs/route.ts — project column present but staff-only
Has a `project` column and could theoretically be filtered per-membership, but was correctly classified staff-only. The audit log is the security accountability trail for Triarch staff actions. The "has project column → project-list" heuristic does not override the security concern here.
