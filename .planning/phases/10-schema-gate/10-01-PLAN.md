**Step D — Add the CHECK constraint to the migration SQL.**

The drizzle-generated SQL will contain `CREATE TABLE "release_log_links"`, `CREATE INDEX` statements (×3), and `ALTER TABLE "projects" ADD COLUMN` statements (×2). Append the CHECK constraint at the end of the file as a separate statement using the multi-statement breakpoint pattern from 0014:

```sql
--> statement-breakpoint
ALTER TABLE "release_log_links" ADD CONSTRAINT "release_log_links_link_type_discriminant"
  CHECK (
    (link_type = 'bug' AND bug_id IS NOT NULL AND feature_id IS NULL AND external_url IS NULL)
    OR (link_type = 'feature' AND feature_id IS NOT NULL AND bug_id IS NULL AND external_url IS NULL)
    OR (link_type = 'external' AND external_url IS NOT NULL AND bug_id IS NULL AND feature_id IS NULL)
  );
```

Why this exact predicate (per CONTEXT.md Implementation Decisions):
- `link_type='bug'` requires `bug_id IS NOT NULL` and the other two discriminant columns NULL
- `link_type='feature'` requires `feature_id IS NOT NULL` and the other two NULL
- `link_type='external'` requires `external_url IS NOT NULL` and both bug_id and feature_id NULL
- This rejects rows that try to link a bug AND a feature on the same row (a bug fix that also closes a feature is two rows, not one), and rejects rows where link_type doesn't match the populated discriminant column

CockroachDB supports multi-column CHECK constraints with logical operators (PostgreSQL-compatible). The constraint is named explicitly so future migrations can DROP/ADD it cleanly.
