<objective>
Create the `PORTAL_NEXTAUTH_SECRET` GCP secret in `triarch-vault` and grant secretAccessor IAM to the (yet-to-be-created) `portal-prod` and `portal-dev` Firebase App Hosting runtime service accounts.

Purpose: Pitfall 1 (catastrophic cross-app JWT replay) requires a distinct NextAuth signing secret per app. This is the storage + IAM half of that requirement. Phase 18 wires the secret into portal's `apphosting.yaml`.
Output: One new GCP secret with one initial version + IAM bindings ready for Plan 15-04 to attach FAH backends to.

Order rationale: This plan can run BEFORE Plan 15-04 (FAH backends) because Firebase App Hosting compute SAs follow a deterministic naming pattern: `service-<project-number>@gcp-sa-firebaseapphosting.iam.gserviceaccount.com` (FAH service agent — single account per project, not per backend) and per-backend compute SAs follow the pattern published in apphosting docs. We grant accessor to the FAH service agent + the project's default Compute Engine SA — the same pattern admin uses for its existing secrets.
</objective>

  <action>
    Step 1 — Create the secret container:

    ```bash
    gcloud secrets create PORTAL_NEXTAUTH_SECRET \
      --project=triarch-vault \
      --replication-policy=automatic \
      --labels=app=portal,owner=mike,phase=15
    ```

    Step 2 — Add an initial version with a 32-byte random base64 payload:

    ```bash
    openssl rand -base64 32 | \
      gcloud secrets versions add PORTAL_NEXTAUTH_SECRET \
        --project=triarch-vault \
        --data-file=-
    ```

    `--data-file=-` reads from stdin so the secret value never lands on disk in plaintext (no `echo $VALUE > /tmp/secret`).

    Step 3 — Mirror admin's IAM bindings on the new secret. For each member discovered in Task 1's secretAccessor binding, run:

    ```bash
    gcloud secrets add-iam-policy-binding PORTAL_NEXTAUTH_SECRET \
      --project=triarch-vault \
      --role=roles/secretmanager.secretAccessor \
      --member='<member from Task 1>'
    ```

    DO NOT add bindings beyond what admin's existing secret has — least-privilege principle. If Task 1 found additional members specific to admin (e.g., `mike@triarchsecurity.com` as a viewer), do NOT mirror those; only mirror SA members.
  </action>
