<objective>
Create two new Firebase App Hosting backends in the `triarch-dev-website` project: `portal-prod` (live branch = `main`, custom domain = `portal.triarch.dev`) and `portal-dev` (live branch = `dev`, FAH-auto domain). Both connect to `MyAlterLego/triarch-portal` (Plan 15-01) and use the runtime SAs that have access to `PORTAL_NEXTAUTH_SECRET` (Plan 15-03).

Purpose: Without backends, the portal repo has nowhere to deploy. Phase 16's first commit (a 200-OK landing page) needs an FAH target to validate the full pipeline. The custom domain wire-up here is what makes `https://portal.triarch.dev` serve a 200-OK landing page once Phase 16 ships.
Output: Two FAH backends visible in `firebase apphosting:backends:list --project triarch-dev-website` with `portal-prod` mapped to `portal.triarch.dev`.

Probable HUMAN-NEEDED step: `firebase` CLI auth may be expired (per orchestrator note). If `firebase login:ci` or `firebase apphosting:backends:create` fails with auth error, fall back to GCP Console manual creation with explicit step-by-step instructions for Mike.
</objective>

<reference>
Existing admin FAH backends in `triarch-dev-website` (precedent — verify with):
```bash
firebase apphosting:backends:list --project triarch-dev-website
```
Expected entries: `triarch-dev` (admin dev), `triarch-dev-prod` or `triarch-prod` (admin prod). Mirror that naming convention with `portal-prod` and `portal-dev`.

Required inputs for `firebase apphosting:backends:create`:
- `--project triarch-dev-website` (Firebase project ID)
- `--location us-central1` (mirror admin; FAH region)
- Backend ID: `portal-prod` or `portal-dev`
- App display name: same as backend ID
- GitHub repo: `MyAlterLego/triarch-portal`
- Live branch: `main` (prod) or `dev` (dev)
- Service account: default FAH compute SA (Phase 24 introduces per-repo deploy SA `portal-deployer@...`)
- Root directory: `/`

GitHub App connection: FAH uses the "Firebase App Hosting" GitHub App. If this is the first time the App is connecting MyAlterLego org, an OAuth-style approval flow runs in browser — `firebase apphosting:backends:create` will print a URL and wait. **This is a HUMAN-NEEDED handoff**: only Mike can approve the App install in MyAlterLego org settings.

Custom domain wire-up sequence (post-creation, only for portal-prod):
1. Backend created → FAH publishes a default `<random>--portal-prod.<region>.hosted.app` hostname
2. Run `firebase apphosting:backends:update portal-prod --domain portal.triarch.dev` (or use Console)
3. FAH issues a TXT challenge for DNS validation; if Plan 15-02 already published the A/CNAME pointing at the FAH IP/host, the challenge auto-passes
4. Managed cert provisions automatically (~10-30 minutes)

If `firebase` CLI is broken (auth expired or apphosting subcommand missing), fall back to:
- Firebase Console → App Hosting → "Get started" / "Add backend"
- Same sequence: project triarch-dev-website → location us-central1 → repo MyAlterLego/triarch-portal → branch main → backend ID portal-prod
- Repeat for portal-dev with branch=dev

GCP gcloud alternative: as of late 2025, `gcloud firebase apphosting backends create` exists in beta. Try `gcloud beta firebase apphosting backends create portal-prod --project=triarch-dev-website ...` if `firebase` CLI fails.
</reference>
