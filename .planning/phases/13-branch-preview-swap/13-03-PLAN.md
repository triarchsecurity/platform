<objective>
Build the customer-facing UI that wires PREV-02 (Preview button on each RC), PREV-04 (in-flight banner + competing buttons disabled), and the client side of PREV-05 (5-second SWR polling, terminal-state pause).

This plan ships a **working but Phase-14-pending integration**: per CONTEXT.md, "Phase 13 ships this as a working but Phase-14-pending integration; Phase 14 polishes layout + adds the swap button to branch section headers." So BranchPreviewClient is rendered as a top-of-list slot in ReleasesClient — visible, functional, complete on its own — but NOT yet woven into the BranchSection headers. That visual polish + per-section button placement is Phase 14's PREV-related work (CUST-03 specifically).

The client island itself is a single component. It owns:
1. The SWR poll (`useSWR` against the GET status route at 5s with `refreshInterval: data?.terminal ? 0 : 5000`)
2. The branch list (passed in from ReleasesClient — derived from `initialSections.map(s => s.branch)` so we don't re-query)
3. The POST dispatcher (fetch wrapper + mutate-on-success)
4. The banner + disabled-button rendering when state ∈ {PENDING, BUILDING, DEPLOYING}
5. The terminal-state pills (SUCCEEDED brief flash, FAILED with FAH console link, timeout)
6. Toast surfaces for 400/409/502 errors

A human-verify checkpoint follows the implementation tasks because the lock-banner copy, button position, FAH console link UX, and 5-second polling cadence all need a real-eye review on a deployed branch before Phase 14 inherits the component. This is exactly the same pattern v2.0 used for customer-facing UX — automation first, verification at the end (per checkpoints section in planner role).

Authorization on the UI side is `userRole === 'admin'` (staff get this everywhere via the existing isStaff override). Viewers see the lock banner if a swap is in flight (informational) but no Preview buttons at all — same authorization granularity already used for the existing Approve button on this page.

Why SWR refreshInterval `data?.terminal ? 0 : 5000`: the GET status route returns `terminal: true` for {idle, SUCCEEDED, FAILED, timeout, CANCELLED}. Setting refreshInterval to 0 in those states pauses the poll — saving the 12 RPS we'd otherwise generate across an idle customer page. SWR resumes when the user clicks a Preview button (we call `mutate()` after a successful POST, which fires one immediate re-fetch and SWR sees `terminal: false` and starts polling again).

Purpose: PREV-02, PREV-04, PREV-05 (client side) — the visible, clickable, polling UI customers actually use to drive a branch swap.
Output: A working customer release page with one new SWR-driven island; a checkpoint confirming the UX before Phase 14 inherits it.
</objective>

'use client';
import useSWR from 'swr';

const fetcher = async (url: string): Promise<StatusResponse> => {
  const res = await fetch(url, { credentials: 'same-origin' });
  if (!res.ok) throw new Error(`status fetch failed: ${res.status}`);
  return res.json();
};

const { data, error, mutate } = useSWR<StatusResponse>(
  `/api/projects/${projectSlug}/branch/preview/status`,
  fetcher,
  {
    refreshInterval: (latest) => (latest?.terminal ? 0 : 5000),
    revalidateOnFocus: false,
    revalidateOnReconnect: true,
    dedupingInterval: 2000,
  },
);
```

Notes:
- `refreshInterval` accepts a function form that receives the latest data — perfect for terminal pause
- `dedupingInterval: 2000` prevents double-fetch when user clicks Preview and component also re-renders
- After POST success, call `mutate()` (no args) to force re-fetch — picks up the new lock immediately

    'use client';
    import React, { useState } from 'react';
    import useSWR from 'swr';
    import { GitBranch, Loader2, AlertCircle, CheckCircle2, ExternalLink } from 'lucide-react';
    import Toast, { type ToastKind } from '@/components/Toast';
    import { formatRelativeTime } from './format';
    import type { UserRole } from './types';

    type StatusResponse = {
      branch: string | null;
      state: 'idle' | 'PENDING' | 'BUILDING' | 'DEPLOYING' | 'SUCCEEDED' | 'FAILED' | 'CANCELLED' | 'timeout';
      locked_at: string | null;
      locked_by: string | null;
      started_at: string | null;
      terminal: boolean;
      errorMessage?: string;
      rolloutResourcePath?: string;
    };

    type Props = {
      projectSlug: string;
      userRole: UserRole;
      branches: string[];
      fahProjectId: string | null;
    };

    const IN_FLIGHT_STATES = new Set(['PENDING', 'BUILDING', 'DEPLOYING']);

    const fetcher = async (url: string): Promise<StatusResponse> => {
      const res = await fetch(url, { credentials: 'same-origin' });
      if (!res.ok) throw new Error(`status fetch failed: ${res.status}`);
      return res.json();
    };

    export default function BranchPreviewClient({ projectSlug, userRole, branches, fahProjectId }: Props) {
      const [dispatching, setDispatching] = useState<string | null>(null);
      const [toast, setToast] = useState<{ kind: ToastKind; message: string; key: number } | null>(null);

      const { data, mutate } = useSWR<StatusResponse>(
        `/api/projects/${projectSlug}/branch/preview/status`,
        fetcher,
        {
          refreshInterval: (latest) => (latest?.terminal ? 0 : 5000),
          revalidateOnFocus: false,
          revalidateOnReconnect: true,
          dedupingInterval: 2000,
        },
      );

      const inFlight = data && IN_FLIGHT_STATES.has(data.state);
      const isAdmin = userRole === 'admin';

      async function handlePreview(branch: string) {
        setDispatching(branch);
        try {
          const res = await fetch(`/api/projects/${projectSlug}/branch/preview`, {
            method: 'POST',
            credentials: 'same-origin',
            headers: { 'Content-Type': 'application/json' },
            body: JSON.stringify({ branch }),
          });
          const payload = await res.json().catch(() => ({}));
          if (res.status === 202) {
            mutate();  // force immediate re-fetch — picks up the new lock
          } else if (res.status === 400) {
            setToast({ kind: 'error', message: 'Branch name not allowed', key: Date.now() });
          } else if (res.status === 409) {
            setToast({ kind: 'error', message: 'Another preview is already in flight', key: Date.now() });
            mutate();  // re-fetch to show the existing lock
          } else if (res.status === 502) {
            setToast({ kind: 'error', message: `Preview dispatch failed${payload.detail ? `: ${payload.detail}` : ''}`, key: Date.now() });
          } else {
            setToast({ kind: 'error', message: 'Could not start preview swap', key: Date.now() });
          }
        } catch {
          setToast({ kind: 'error', message: 'Network error during preview swap', key: Date.now() });
        } finally {
          setDispatching(null);
        }
      }

      // Render: banner (when in-flight or terminal-non-idle) + button row + toast
      // ...exact JSX with Tailwind: zinc dark + amber for in-flight banner + violet for dispatching + red for FAILED + emerald for SUCCEEDED
      // FAH console link: https://console.firebase.google.com/project/${fahProjectId}/apphosting (only when fahProjectId truthy)
      // For each branch in props.branches, render a button (only when isAdmin); disabled when inFlight OR dispatching === branch
    }
    ```
