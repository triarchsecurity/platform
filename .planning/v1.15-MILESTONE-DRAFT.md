# v1.15 — Multi-Branch RC + Central Vault + OttoBot Brain (DRAFT)

**Status:** Stub — to be expanded via `/gsd:new-milestone` in a fresh session.
**Source:** Architectural decisions captured during v1.14 final session, 2026-05-04.
**Headline:** Customer-gated parallel release candidates with auto-rebase-and-merge promotion, unified credential storage, and OttoBot as the canonical Slack control plane.

---

## Why This Milestone

Three architectural needs surfaced during v1.14 closeout that are too big for v1.14 but interlock cleanly:

1. **Multi-branch RC pattern (Mike's primary ask):** v1.14 assumes one release per project at a time on main. Real workflow needs multiple feature branches deploying as parallel preview RCs, customer reviews each independently, approval triggers auto-rebase-and-merge so no prior work gets reverted when a later RC promotes.
2. **Central credential vault:** OttoBot's Slack token currently lives in CRM `settings` table AND in `triarch-dev-website` Firebase secrets after v1.14. Drift risk on rotation. Each project needs centralized access patterns to shared creds (Slack, GitHub Apps).
3. **OttoBot as Slack brain:** OttoBot needs single canonical interactivity URL with a dispatcher that routes by `action_id`. Wider permissions for automated workflow execution. v1.14's basic dispatcher (built last in v1.14 session) becomes permanent fixture, gets audit logging + metrics.

All three share infrastructure (the dispatcher needs the vault for token access; the multi-branch flow uses OttoBot for approve buttons; the vault unblocks shared-workflows clean access). One milestone, intertwined phases.

## Architecture Sketch

### Multi-Branch RC Flow

```
T+T main:                        a───b───c                                   ┐
                                /         \                                  │
feat/change-font (week 1):  ───d─┘  → preview at feat-change-font--...      │
                                  ↓                                          │ Customer
                       Customer approves (Slack click)                       │ reviews
                                  ↓                                          │ each
                       Rebase feat/change-font on main, CI, merge → main     │ branch
                                                                             │
feat/add-audio (week 1):    ───f───────────────g  → preview at              │ in
                                                                feat-add-... │ parallel
                                  ↓                                          │
                       Customer approves audio                               │
                                  ↓                                          │
                       Rebase audio on UPDATED main (now has font)           │
                       CI runs on rebased branch                             │
                       Green → merge. Conflict → Slack "needs manual"        ┘
```

### Architectural Pieces

| # | Piece | Notes |
|---|-------|-------|
| 1 | `release_logs.branch varchar(256)` | New column. RCs key on `(project_id, branch, version)` |
| 2 | Branch preview deploys | Use FAH `apphosting:rollouts:create --git-branch <branch>`. Already native — just wire into `shared-workflows/deploy-firebase.yml` for non-main branches |
| 3 | Customer page grouping | `/projects/{slug}/releases` groups RCs by branch. Shows preview URL + per-RC Approve button. UI gate likely needed |
| 4 | GitHub App permission upgrade | Current `contents:read` → `contents:write` for the merge step |
| 5 | New `promote-branch.yml` workflow | Slack approve dispatches this. Does: `fetch`, `rebase branch onto main`, conflict-detect, `push main` (or `merge --no-ff`) |
| 6 | Conflict notification path | New Slack action: `:warning: Cannot promote {branch} — conflicts with main: {file list}. Rebase manually and re-deploy.` |
| 7 | Phase-4 `promoteAndAudit` rewrite | Dispatches `promote-branch.yml` with `branch` input instead of `deploy-prod.yml` with `tag` |
| 8 | Preview URL surfacing | Page shows `<ExternalLink>` per RC pointing to its preview URL |

### Central Vault

```
GCP project: triarch-vault (new, dedicated)
  Secret Manager:
    - SLACK_BOT_TOKEN, SLACK_SIGNING_SECRET, SLACK_PAYLOAD_SECRET
    - GITHUB_APP_ID, GITHUB_APP_PRIVATE_KEY, GITHUB_APP_INSTALLATION_ID
    - SLACK_USER_MAP (json blob)
    - resend_api_key, gemini_api_key, anthropic_api_key, etc.
        ▲
        │ IAM grants per project's service account
        │
  Firebase projects (admin, CRM, tmi, www, portal, etc.)
    Per-project secrets stay local: DATABASE_URL, NEXTAUTH_SECRET
    Shared secrets pulled from triarch-vault at runtime
```

Implementation: thin npm package `@myalterlego/secrets` wrapping `@google-cloud/secret-manager` + module-level cache. Each project imports `await getSecret('SLACK_BOT_TOKEN')` instead of reading `process.env`. Existing Firebase per-project secrets stay as fallback during migration.

### OttoBot Dispatcher (Permanent Fixture)

Built in v1.14 closing session (basic version). v1.15 adds:
- Structured audit logging — every action_id click writes a row to `slack_action_audit` table
- Metrics — action_id counts, response time, failure rate exported to whatever observability stack you adopt
- OttoBot scope expansion: `chat:write.public`, `app_mentions:read`, `commands` (slash commands)
- Slash command handler: `/triarch deploy <project> <version>` triggers workflow_dispatch (only for staff in SLACK_USER_MAP)
- App mentions: `@OttoBot status <project>` queries release_logs and posts back current dev/prod state
- `slack_action_audit` table for compliance
