**Phase 2 additions:**

## State Matrix

Each status drives what actions and UI elements render in the expanded panel.
The "action buttons row" renders only for users with `role = 'admin'`.

| Status | Approve button | Reject button | Action buttons row border | Audit trail line | Feedback compose |
|--------|----------------|---------------|---------------------------|------------------|------------------|
| `dev` | Visible, step-1 state | Visible, step-1 state | Visible | Hidden | Visible (admin only) |
| `pending_approval` | Hidden | Hidden | Hidden | Hidden | Visible (admin only) |
| `approved` | Hidden | Hidden | Hidden | Visible: "approved by X on Y" | Visible (admin only) |
| `rejected` | Hidden | Hidden | Hidden | Visible: "rejected by X: excerpt" | Visible (admin only) |
| `promoted` | Hidden | Hidden | Hidden | Visible: "approved by X on Y · promoted on Z" | Visible (admin only) |

**Viewer role override:** All compose areas (feedback textarea, reject form) are not rendered regardless of status. Action buttons never rendered. Viewer sees: table row, expanded panel with feedback list + audit trail line only.

**Note on `pending_approval`:** This status is reserved for Phase 3 (Slack-driven approval). In Phase 2, approval happens directly in the UI — status transitions `dev → approved` or `dev → rejected`. The `pending_approval` state will be introduced in Phase 3 when Slack buttons are wired. For Phase 2, treat `pending_approval` as a terminal display-only state with no action buttons.

## Two-Step Approve Interaction

**Step 1 — Initial button:**
```tsx
<button
  onClick={() => setApproveStep('confirm')}
  className="flex items-center gap-1.5 px-3 py-2 text-sm bg-teal-600 text-white
             rounded-md hover:bg-teal-500"
>
  <CheckCircle size={14} /> Approve for Production
</button>
```

**Step 2 — Countdown (5 seconds):**

On click: transition button to confirmation state, start 5-second countdown using `useEffect` + `setInterval`. Text countdown only — no progress ring. No separate cancel button.

```tsx
<button
  onClick={() => handleApprove(releaseId)}
  className="flex items-center gap-1.5 px-3 py-2 text-sm bg-teal-700 text-white
             rounded-md border border-teal-500 min-w-[160px] justify-center"
>
  Confirm approval ({countdown}s…)
</button>
```

**Cancel behavior:** Click anywhere outside the button area OR timer reaching 0 without a second click resets to Step 1. Implement via `onBlur` on the container div or a document-level click listener. If countdown expires without confirmation, revert to Step 1 with no API call.

**Accessibility:**
- `aria-label="Confirm release approval"` on Step 2 button.
- `aria-live="polite"` on the countdown text so screen readers announce the change.
- Focus moves to the Step 2 button automatically on transition (call `.focus()` after state update).
- Tab order: Approve button → Reject button → Post Comment button. Action buttons come after the feedback compose area in DOM order.

      <button
        onClick={() => { setRejectReason(''); setShowRejectForm(false); }}
        className="text-xs text-zinc-500 hover:text-zinc-300 px-2 py-1.5"
      >
        Cancel
      </button>
      <button
        onClick={() => handleReject(releaseId)}

- Character limit: 500 chars. Server enforces; client warns at 450.
- `autoFocus` on the textarea so the user can immediately type.

- Empty/whitespace-only submit: button disabled (`!rejectReason.trim()`). No server call.
- Focus returns to the Reject Release button if user cancels.

## Toast Notification

**Implementation:** Hand-rolled — no external library. A single `<Toast>` component fixed to `bottom-right`. One toast visible at a time; subsequent toasts replace the previous.

**Position:** `fixed bottom-6 right-6 z-50`

**Duration:** 5 seconds auto-dismiss. Success toasts: 5s. Error toasts: persistent until dismissed (no auto-dismiss on error — user may need to read it).

```tsx
<div
  role="status"
  aria-live="polite"
  className={`fixed bottom-6 right-6 z-50 w-80 px-4 py-3 rounded-lg border shadow-lg
              text-sm transition-all ${
    kind === 'success'
      ? 'bg-zinc-900 border-teal-500/30 text-zinc-200'
      : 'bg-zinc-900 border-red-500/30 text-red-400'
  }`}
>
  <div className="flex items-start justify-between gap-3">
    <span>{message}</span>
    <button onClick={dismiss} aria-label="Dismiss notification" className="text-zinc-600 hover:text-zinc-400 flex-shrink-0">
      <X size={14} />
    </button>
  </div>
</div>
```

- `role="status"` + `aria-live="polite"` for screen reader announcement.
- Success: dark zinc card with teal border. Not a green banner — avoids visual clash with the in-page feedback banners.
- Error: dark zinc card with red border + red text.
- No progress bar. No duration indicator. Just text + X dismiss.

**In-place row update:** After approve/reject, the toast fires AND the release row updates in local state — status badge changes, action buttons disappear, audit trail line appears. No full page refetch.

## Interaction Contracts

**Feedback delete — no confirmation modal:**
Clicking the Trash2 icon on a comment fires the delete immediately with no confirmation dialog. Rationale: comments are scoped to the author and deletable only within 24 hours. The 24-hour window is the sole safeguard. A "Comment deleted." toast fires on success. This mirrors the existing provisioning UX in the project list where destructive row actions are immediate with a toast. No undo. No confirmation step.

## Accessibility Contract

| Element | Requirement |
|---------|-------------|
| Table expand toggle | `aria-expanded={expanded}` + `aria-controls="panel-{releaseId}"` on the `<tr>` click handler or a button within it |
| Expanded panel | `id="panel-{releaseId}"` on the `<tr>` containing the expanded content |
| Status badge | `aria-label="{status} status"` — badge text alone may be ambiguous |
| Approve button step 1 | `aria-label="Approve release {version} for production"` |
| Approve button step 2 | `aria-label="Confirm release approval"` + `aria-live="polite"` on countdown text |
| Reject button | `aria-label="Reject release {version}"` |
| Reject form textarea | `aria-required="true"` + `aria-label="Rejection reason"` |
| Feedback textarea | `aria-label="Leave a comment"` |
| Post Comment button | `aria-label="Post comment for release {version}"` |
| Delete comment | `aria-label="Delete comment"` |
| Toast | `role="status"` + `aria-live="polite"` |
| Toast dismiss button | `aria-label="Dismiss notification"` |
| Error banner | `role="alert"` + `aria-live="assertive"` |
| Skeleton rows | `aria-busy="true"` on `<tbody>` during loading |

**Focus management:**
- When the approve button transitions to step 2, call `.focus()` on the new button element.
- When the reject form opens, `autoFocus` on the textarea handles initial focus.
- When the reject form is cancelled, return focus to the Reject Release button.
- When a feedback comment is deleted, return focus to the first remaining comment or the textarea.

**Role gating in markup:**
- Action buttons and feedback compose textarea are conditionally rendered based on `userRole`.
- Do NOT render with `disabled` and hide via CSS — elements must not be in the DOM for non-admin users (viewer role). Screen readers should not encounter hidden action elements.
- Server component passes `userRole: 'admin' | 'viewer'` as a prop to the client component.
