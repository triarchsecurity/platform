<objective>
Implement approve + reject API endpoints that atomically (a) insert a `release_approvals` audit row and (b) update `release_logs.status`. Idempotent re-approval is a no-op (returns 200 with existing row); rejection requires a non-empty reason; rejected releases cannot be re-approved (409).

Purpose: This is the core gating action — covers GATE-04 (admin-only + status precondition), GATE-05 (audit row capturing approver/timestamp/IP/UA + idempotency), GATE-06 (atomic dev to approved), and REJECT-01 (reject with reason, dev to rejected, non-re-approvable). Plans 02 + 05 wire the UI to these endpoints; this plan owns the server logic.

Output: Two route handlers using `db.transaction()` for atomicity. Headers captured from the request. Role-gated to admin (or staff bypass).
</objective>

Export `POST(req, { params }: { params: Promise of { slug, releaseId } })`:

1. Call `requireSignedIn()`. If error, return error.
2. Await params; call `getCurrentUserContext(session)`. If null, return 403 Forbidden.
3. Look up project: `db.select({ key: projects.key }).from(projects).where(eq(projects.key, slug))`. If not found, return 404 Not found.
4. Find membership: `ctx.memberships.find(m => m.project_key === project.key)`. Set `isMember = ctx.isStaff || !!membership`. If not member, return 404 Not found (no leak).
5. Set `isAdmin = ctx.isStaff || membership?.role === 'admin'`. If not admin, return 403 Forbidden.
6. Look up release: `db.select().from(releaseLogs).where(and(eq(releaseLogs.id, releaseId), eq(releaseLogs.project, project.key)))`. If not found, return 404.
7. **Idempotent short-circuit**: if `release.status === 'approved'`, look up the most recent approved row for this release (orderBy desc(approvedAt) limit 1) and return JSON `{ ok: true, alreadyApproved: true, release: { id, status }, approval: <serialised row or null> }` with status 200. Do NOT call db.transaction in this branch.
8. Status precondition: `currentStatus = release.status ?? 'dev'`. If `currentStatus !== 'dev'`, return JSON `{ error: "Cannot approve a release in status '<currentStatus>'" }` status 409.
9. Capture audit context:
   - `ipAddress = req.headers.get('x-forwarded-for')?.split(',')[0]?.trim() ?? null`
   - `userAgent = req.headers.get('user-agent')?.slice(0, 512) ?? null`
10. Call `db.transaction(async (tx) => { ... })`:
    - Inside the tx callback, INSERT into releaseApprovals with values `{ releaseId: release.id, approverEmail: ctx.email, decision: 'approved', ipAddress, userAgent, reason: null }`. Use `.returning()`. Capture as `inserted`.
    - Then UPDATE releaseLogs SET status='approved' WHERE id=release.id, returning `{ id: releaseLogs.id, status: releaseLogs.status }`. Capture as `updated`.
    - Return `{ inserted, updated }` from the callback.
11. Return JSON with status 200:
    `{ ok: true, alreadyApproved: false, release: { id: result.updated.id, status: result.updated.status }, approval: { id, releaseId, approverEmail, decision, approvedAt: <ISO string>, reason, ipAddress, userAgent } }`

Key invariants (must hold):
- approverEmail is taken from session via ctx.email — NEVER from request body (security).
- 404 (not 403) for non-members and for releases in another project — no leak.
- 403 for member-but-not-admin — distinguishable.
- The release.status === 'approved' branch fires BEFORE the precondition gate so re-approve is idempotent (GATE-05).
- Only dev to approved is allowed; rejected/pending_approval/promoted return 409.
- IP/UA truncated to schema column widths (varchar 45 / varchar 512).
- Response includes `alreadyApproved: boolean` so the client picks the correct toast (UI-SPEC line 651: "This release was already approved.").
