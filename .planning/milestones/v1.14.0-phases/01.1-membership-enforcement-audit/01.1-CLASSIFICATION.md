## Special Case Notes

### reports/[id]/route.ts — DELETE caution
Row 28 is classified `project-detail` but the DELETE handler should be **staff-only** within Plan 05's implementation. The classification drives the access check shape (fetch row first, check membership), but Plan 05 should additionally require `ctx.isStaff` for DELETE — non-staff customers should read and comment on reports, not delete them. This is noted here so Plan 05's implementer doesn't accidentally open delete to any project member.

### projects/route.ts — dual behavior
Row 18 is classified `project-list` because the dominant GET behavior is membership-filtered. However POST (project creation) must be wrapped with `requireStaff()` inside Plan 04's implementation — creating a project has no customer use case. Plan 04 should branch on method: GET uses `requireSignedIn` + existing filter; POST uses `requireStaff`.

### access-logs/route.ts — project column present but still staff-only
Row 1 has a `project` column and could theoretically be filtered per-membership. However, access audit logs are the security accountability trail for Triarch staff actions — customers must never see who accessed what data across the platform even for their own project. This classification overrides the naive "has project column → project-list" heuristic.
